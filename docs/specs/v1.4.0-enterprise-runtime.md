## 1. Problem Statement

SyndicateClaw v1.3.0 can coordinate multi-agent workflows, but lacks enterprise features: workflows must be triggered manually, there is no multi-tenant isolation, and performance under load is unknown. Organizations deploying SyndicateClaw need scheduled workflows, per-org isolation, and confidence in throughput.

---

## 2. Goals

- **Scheduled workflows**: Workflows can run on cron, interval, or one-time schedules
- **Multi-tenancy**: Organizations with isolated namespaces, RBAC role mappings, and quotas
- **Performance**: Load-tested to documented throughput targets with realistic workflows

---

- SDK (v1.5.0)

## 4. Scheduled Workflows

### 4.1 Schedule Model

```python
class WorkflowSchedule(BaseModel):
    id: str                      # ULID
    workflow_id: str             # Reference to workflow definition
    workflow_version: int | None # Pin to version (None = latest)
    name: str                    # Human-readable name
    description: str | None
    schedule_type: ScheduleType  # CRON | INTERVAL | ONCE
    schedule_value: str          # Cron expression, pytimeparse duration ("1h", "30m"), or ISO datetime
    input_state: dict[str, Any]  # Initial state for each triggered run
    actor: str                   # Actor who created the schedule (set server-side)
    namespace: str               # Organization namespace
    status: ScheduleStatus       # ACTIVE | PAUSED | COMPLETED | FAILED
    next_run_at: datetime
    last_run_at: datetime | None
    max_runs: int | None         # Stop after N runs (None = infinite)
    run_count: int               # Runs executed so far
    # Lock fields for HA duplicate-execution prevention
    locked_by: str | None        # Scheduler instance ID that holds the current lock
    locked_until: datetime | None # Lock expiry; another instance may claim after this time
    created_at: datetime
    updated_at: datetime
```

### 4.2 Schedule Types

| Type | `schedule_value` format | Example | Parser |
|------|------------------------|---------|--------|
| `CRON` | Standard 5-field cron expression | `0 */6 * * *` (every 6 hours) | `croniter` |
| `INTERVAL` | Duration string | `30m`, `1h`, `2d` | `pytimeparse` |
| `ONCE` | ISO 8601 datetime | `2026-04-01T09:00:00Z` | `datetime.fromisoformat` |

Schedule values are validated at creation time using the appropriate parser. Invalid expressions return HTTP 400 with a descriptive error.

### 4.3 Scheduler Service

An asyncio-based background service. Uses PostgreSQL `SELECT FOR UPDATE SKIP LOCKED` for efficient, contention-free schedule pickup under concurrent scheduler instances.

```python
class SchedulerService:
    instance_id: str  # Unique ID per scheduler instance (set at startup, e.g., hostname + PID)

    async def run(self):
        while True:
            await self._process_due_schedules()
            await asyncio.sleep(self.poll_interval_seconds)

    async def _process_due_schedules(self):
        async with db_session() as session:
            # SKIP LOCKED: other scheduler instances skip rows locked by this one.
            # This eliminates hot-loop contention under concurrent schedulers.
            due = await session.execute(
                """
                SELECT * FROM workflow_schedules
                WHERE status = 'ACTIVE'
                  AND next_run_at <= NOW()
                  AND (locked_by IS NULL OR locked_until < NOW())
                ORDER BY next_run_at
                LIMIT :batch_size
                FOR UPDATE SKIP LOCKED
                """,
                {"batch_size": self.batch_size}
            )
            for schedule in due.fetchall():
                await self._claim_and_execute(session, schedule)

    async def _claim_and_execute(self, session, schedule):
        # Atomic claim: set lock fields within the same transaction as the SELECT FOR UPDATE
        lock_expiry = datetime.utcnow() + timedelta(seconds=self.lock_lease_seconds)
        await session.execute(
            """
            UPDATE workflow_schedules
            SET locked_by = :instance_id, locked_until = :lock_expiry
            WHERE id = :id
            """,
            {"instance_id": self.instance_id, "lock_expiry": lock_expiry, "id": schedule.id}
        )
        # Commit the lock before spawning the task
        await session.commit()
        asyncio.create_task(self._execute_schedule(schedule))
```

**Lock lease duration**: `SYNDICATECLAW_SCHEDULER_LOCK_LEASE_SECONDS` (default: 120 seconds). If a scheduler instance crashes after claiming a lock, the lock expires after this interval and another instance picks up the schedule on its next poll.

**Execution and lock release**: After a successful or failed run creation, the scheduler updates `next_run_at`, `last_run_at`, `run_count`, and clears `locked_by`/`locked_until` in a single transaction. If run creation itself fails, the lock is also cleared so the schedule is retried on the next poll.

### 4.4 Concurrency

- Multiple scheduler instances can run for HA. `SKIP LOCKED` ensures each schedule is processed by exactly one instance per tick.
- `SYNDICATECLAW_SCHEDULER_MAX_CONCURRENT` (default: 50) limits concurrent in-flight schedule executions per instance.
- The scheduler respects the platform-level `SYNDICATECLAW_MAX_CONCURRENT_RUNS` limit. If the global limit is reached, the scheduler defers the scheduled run and retries on the next poll, logging `scheduler.deferred_run_limit_reached`.

## 5. Multi-Tenancy

### 5.1 Design Philosophy

v1.4.0 implements **namespace-scoped multi-tenancy** at the application layer. All resource tables gain a non-nullable `namespace` column. PostgreSQL row-level security is not enforced at the database layer in this release (acknowledged residual risk per the threat model). Application-level namespace filtering must be applied consistently on every query.

This is accurately described as **namespace-scoped isolation**, not full database-layer multi-tenancy. The distinction matters for compliance conversations.

### 5.2 Organization Model

```python
class Organization(BaseModel):
    id: str                      # ULID
    name: str                    # Unique name (slug, e.g., "acme-corp")
    display_name: str
    owner_actor: str             # Actor string of the creating user (consistent with platform's actor model)
    namespace: str               # Derived from name; used as namespace on all resources
    status: OrganizationStatus   # ACTIVE | SUSPENDED | DELETING
    quotas: OrganizationQuotas
    settings: dict[str, Any]
    created_at: datetime
    updated_at: datetime

class OrganizationQuotas(BaseModel):
    rate_limit_requests: int = 1000     # Requests per hour
    rate_limit_burst: int = 100         # Burst per second
    max_agents: int = 50
    max_workflows: int = 200
    max_schedules: int = 100
    max_memory_records: int = 100000
    storage_limit_bytes: int = 10_737_418_240  # 10GB — see §5.6 for enforcement
```

### 5.3 Namespace Isolation

Every resource table gains a `namespace TEXT NOT NULL` column. The migration uses a **NOT NULL column with an inline DEFAULT `'default'` for the migration only** (see §7). After the migration, the DEFAULT is removed and `namespace` is enforced as required on all new inserts.

Query pattern — every list/search query includes a namespace filter:

```python
# Enforced at the repository layer, not the route handler
async def list_workflows(actor_namespace: str) -> list[WorkflowDefinition]:
    return await session.execute(
        "SELECT * FROM workflow_definitions WHERE namespace = :ns",
        {"ns": actor_namespace}
    )
```

**Test requirement**: A dedicated integration test suite (`tests/test_namespace_isolation.py`) asserts that every repository method filters by namespace. The CI gate fails if any query method does not include namespace in its WHERE clause (enforced via a code review checklist and optional AST linter rule).

### 5.4 Organization Membership and Role Mapping

```python
class OrganizationMember(BaseModel):
    organization_id: str
    actor: str                  # Actor string (consistent with platform actor model)
    org_role: OrgRole           # OWNER | ADMIN | MEMBER | VIEWER
    rbac_role: str              # Mapped RBAC role for permission resolution (see table below)
    joined_at: datetime
```

**OrgRole → RBAC role mapping:**

| OrgRole | Maps to RBAC Role | Rationale |
|---------|-------------------|-----------|
| `OWNER` | `tenant_admin` | Full org control including team/quota management |
| `ADMIN` | `admin` | Full resource access within org namespace |
| `MEMBER` | `operator` | Read/write workflows, agents, memory |
| `VIEWER` | `viewer` | Read-only access |

When an actor authenticates, their `org_role` is looked up from `organization_members` and mapped to the corresponding RBAC role. This mapping is the **single source of truth** for permissions. The JWT does not carry `permissions` or expanded role grants (see §5.5).

### 5.5 Auth Integration

API keys and JWTs carry identity and organization context **only**. Permissions are resolved at request time from the RBAC system using the identity claims — they are not embedded in the token.

```json
{
  "sub": "mikeholownych",
  "org_id": "org-abc123",
  "org_role": "OWNER",
  "exp": 1743000000
}
```

**What is NOT in the JWT**: `permissions` array. Embedding permissions in JWTs:
- Makes permission revocation require waiting for token expiry (hours)
- Creates a second, potentially stale source of truth that conflicts with the live RBAC system
- Bloats token size for actors with many permissions

The `org_role` claim is a coarse hint used to skip a DB lookup in the common case, but the authoritative permission check is always performed against the live RBAC system.

### 5.6 Quota Enforcement

Quotas are enforced via **route-level quota decorators**, not middleware path-string matching.

```python
# Route-level decorator — applied per endpoint, not inferred from path
@router.post("/api/v1/workflows")
@require_quota("max_workflows", get_workflow_count)
async def create_workflow(actor_org: Organization = Depends(get_actor_org), ...):
    ...
```

`@require_quota(quota_field, count_fn)`:
1. Resolves `actor_org.quotas.<quota_field>`.
2. Calls `count_fn(org_id)` to get the current count.
3. If count >= limit, returns HTTP 429 with body `{"error": "quota_exceeded", "quota": "<field>", "limit": <n>, "current": <n>}`.

**All quota types and their enforcement points:**

| Quota | Enforced On |
|-------|-------------|
| `max_workflows` | `POST /api/v1/workflows` |
| `max_agents` | `POST /api/v1/agents` |
| `max_schedules` | `POST /api/v1/workflows/{id}/schedule` |
| `max_memory_records` | `POST /api/v1/memory` |
| `rate_limit_requests` / `rate_limit_burst` | `RateLimitMiddleware` (existing, scoped to `org_id`) |
| `storage_limit_bytes` | `POST /api/v1/memory` — byte size of `value` field accumulated per org. Count tracked in `organization_quotas_usage` table, updated on write, checked before insert. |

### 5.7 Cross-Namespace Access

Actors with `admin:*` permission cannot silently bypass namespace filtering. Cross-namespace access requires an **explicit impersonation session** (defined in the RBAC design spec) with:

- Audit event: `admin.cross_namespace_access actor=<> target_namespace=<> reason=<>`
- Impersonation session logged as a first-class record
- All actions during the session attributed to `real_actor=<admin>` + `effective_namespace=<target>`

This applies to platform-level support operations. Org-level OWNER/ADMIN actors only see their own namespace.

### 5.8 Organization Deletion Lifecycle

When an organization transitions to `DELETING` status:

1. **Immediately**: All `ACTIVE` schedules for the org are `PAUSED`. New runs cannot be created (quota check blocks with HTTP 409 `org_deleting`).
2. **In-flight runs** (`RUNNING`, `WAITING_APPROVAL`, `WAITING_AGENT_RESPONSE`): Allowed to complete or timeout. Status is checked every 60 seconds.
3. **Background cleanup job** (`syndicateclaw.tasks.org_cleanup`): Once all in-flight runs are terminal, the job soft-deletes all org resources in dependency order: runs → workflows → agents → memory → messages → schedules → members → organization. Each phase is a separate audited transaction.
4. **Completion**: Organization transitions to `DELETED`. Namespace is released and may be reused after 30-day hold period.

### 6.2 Workflow Run State Caching

Cache the **state dict only** (not the full `WorkflowRun` ORM object) for active runs in Redis. The cache key is `syndicateclaw:run_state:{run_id}`.

```python
class StateCache:
    # TTL varies by run status to prevent unbounded Redis growth
    TTL_BY_STATUS: dict[str, int] = {
        "RUNNING": 3600,
        "WAITING_APPROVAL": 3600,
        "WAITING_AGENT_RESPONSE": 3600,
        "COMPLETED": 60,    # Short TTL: run is done; state still briefly useful
        "FAILED": 60,
        "CANCELLED": 60,
        "PAUSED": 600,
    }

    async def get(self, run_id: str) -> dict | None:
        cached = await self.redis.get(f"syndicateclaw:run_state:{run_id}")
        return json.loads(cached) if cached else None

    async def set(self, run_id: str, state: dict, status: str):
        ttl = self.TTL_BY_STATUS.get(status, 300)
        await self.redis.setex(
            f"syndicateclaw:run_state:{run_id}",
            ttl,
            json.dumps(state)
        )

    async def invalidate(self, run_id: str):
        await self.redis.delete(f"syndicateclaw:run_state:{run_id}")
```

Cache is invalidated on every state update (write-through) and on run status transitions to terminal states.

**Namespace migration note**: Each migration is a separate Alembic revision so that individual table changes can be rolled back independently. Each migration:
1. Adds `namespace TEXT DEFAULT 'default'` (temporarily nullable-compatible default).
2. Runs `UPDATE <table> SET namespace = 'default' WHERE namespace IS NULL`.
3. Alters the column to `NOT NULL` (removing the default).

This ensures no row has a NULL namespace after migration. The NOT NULL constraint is enforced from v1.4.0 forward.

## 12. Rollback

- **Scheduling**: Disable scheduler service (`SYNDICATECLAW_SCHEDULER_ENABLED=false`); `workflow_schedules` table exists but unused.
- **Multi-tenancy**: Namespace columns exist; application namespace filtering can be disabled via `SYNDICATECLAW_ORGANIZATION_ENABLED=false` which disables org enforcement and routes all actors to namespace `"default"`.
- **Performance indexes**: Indexes can be dropped without data loss.
- **State cache**: Cache is optional; Redis failure already degrades gracefully.
