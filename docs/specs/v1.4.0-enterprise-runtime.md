# v1.4.0 — Enterprise Runtime

**Sprint Duration:** 3 weeks
**Goal:** Add scheduling, multi-tenancy, and performance hardening

---

## 1. Problem Statement

SyndicateClaw v1.3.0 can coordinate multi-agent workflows, but lacks enterprise features: workflows must be triggered manually, there's no multi-tenant isolation, and performance under load is unknown. Organizations deploying SyndicateClaw need scheduled workflows, per-org isolation, and confidence in throughput.

## 2. Goals

- **Scheduled workflows**: Workflows can run on cron, interval, or one-time schedules
- **Multi-tenancy**: Organizations with isolated namespaces, RBAC, and quotas
- **Performance**: Load-tested to target throughput with realistic workflows

## 3. Non-Goals

- Visual workflow builder (v1.5.0)
- SDK (v1.5.0)
- Kubernetes operator (future)
- Geographic distribution (future)

## 4. Scheduled Workflows

### 4.1 Schedule Model

```python
class WorkflowSchedule(BaseModel):
    id: str                      # ULID
    workflow_id: str             # Reference to workflow definition
    workflow_version: int | None # Pin to version (None = latest)
    name: str                    # Human-readable name
    description: str | None
    schedule_type: ScheduleType  # CRON | INTERVAL | ONCE
    schedule_value: str          # Cron expression or duration (e.g., "*/5 * * * *" or "1h")
    input_state: dict[str, Any] # Initial state for each run
    actor: str                   # Who created the schedule
    namespace: str               # Organization namespace
    status: ScheduleStatus       # ACTIVE | PAUSED | COMPLETED | FAILED
    next_run_at: datetime
    last_run_at: datetime | None
    max_runs: int | None         # Stop after N runs (None = infinite)
    run_count: int               # Runs executed so far
    created_at: datetime
    updated_at: datetime
```

### 4.2 Schedule Types

| Type | `schedule_value` | Example |
|------|-----------------|---------|
| `CRON` | Cron expression | `0 */6 * * *` (every 6 hours) |
| `INTERVAL` | Duration string | `30m`, `1h`, `2d` |
| `ONCE` | ISO datetime | `2026-04-01T09:00:00Z` |

### 4.3 Scheduler Service

An asyncio-based background service that:

1. **Polls** `workflow_schedules` table for schedules where `next_run_at <= now()`.
2. **Locks** the schedule (optimistic locking via `status` field) to prevent duplicate execution.
3. **Executes** the workflow with `input_state`.
4. **Updates** `next_run_at`, `last_run_at`, `run_count`.
5. **Completes** when `run_count >= max_runs` (for `ONCE` or limited schedules).

```python
class SchedulerService:
    async def run(self):
        while True:
            due_schedules = await self._fetch_due_schedules()
            for schedule in due_schedules:
                if self._try_lock(schedule):
                    asyncio.create_task(self._execute_schedule(schedule))
            await asyncio.sleep(self.poll_interval_seconds)
```

### 4.4 Concurrency

- Multiple scheduler instances can run (for HA). Locking prevents duplicate execution.
- Lock uses a lease mechanism: `locked_by` + `locked_until`. If a scheduler crashes, the lock expires and another instance picks up.
- Max concurrent schedule executions: configurable via `scheduler_max_concurrent`.

## 5. Multi-tenancy

### 5.1 Organization Model

```python
class Organization(BaseModel):
    id: str                      # ULID
    name: str                    # Unique name
    display_name: str
    owner_id: str                # Admin actor
    namespace: str               # Derived from name (e.g., "acme-corp")

    updated_at: datetime

class OrganizationQuotas(BaseModel):
    rate_limit_requests: int = 1000     # Requests per hour
    rate_limit_burst: int = 100         # Burst per second
    max_agents: int = 50                # Max registered agents
    max_workflows: int = 200            # Max workflow definitions
    max_schedules: int = 100            # Max active schedules
    max_memory_records: int = 100000    # Max memory records
    storage_limit_bytes: int = 10737418240  # 10GB default
```

### 5.2 Namespace Isolation

Every resource is scoped to a namespace:

- `workflow_definitions.namespace`
- `workflow_runs.namespace`
- `agents.namespace`
- `memory_records.namespace`
- `policy_rules.namespace` (org-scoped policies)
- `agent_messages.namespace`

All API endpoints enforce namespace scoping:
- `GET /api/v1/workflows` — only returns workflows in the actor's namespace
- `POST /api/v1/workflows` — namespace is derived from the authenticated actor's organization
- Cross-namespace access requires `admin:*` permission

### 5.6 Quota Enforcement

Quotas are enforced at the middleware level:

```python
@middleware
async def quota_middleware(request, call_next):
    org = get_organization(request)
    quota = org.quotas

    # Check rate limit (uses existing Redis-backed limiter, scoped to org)
    if await is_rate_limited(org.id, quota.rate_limit_requests, quota.rate_limit_burst):
        return Response(status_code=429)

    # Check resource limits on creation
    if request.method == "POST" and request.url.path.endswith("/workflows"):
        if await get_workflow_count(org.id) >= quota.max_workflows:
            return Response(status_code=429, content="Workflow quota exceeded")

    return await call_next(request)
```

### 6.2 Workflow State Caching

Cache active workflow run state in Redis:

```python
class StateCache:
    async def get(self, run_id: str) -> dict | None:
        cached = await self.redis.get(f"syndicateclaw:run_state:{run_id}")
        return json.loads(cached) if cached else None

    async def set(self, run_id: str, state: dict, ttl: int = 3600):
        await self.redis.setex(
            f"syndicateclaw:run_state:{run_id}",
            ttl,
            json.dumps(state)
        )

    async def invalidate(self, run_id: str):
        await self.redis.delete(f"syndicateclaw:run_state:{run_id}")
```

Cache is invalidated on:
- State update (write-through)
- Run completion
- Run failure

## 9. Testing Strategy

### 9.1 Scheduling Tests

- `test_schedule_creates_workflow_run` — Schedule fires and creates a run
- `test_schedule_respects_cron` — Cron expression produces correct next_run_at
- `test_schedule_interval` — Interval produces correct next_run_at
- `test_schedule_once_completes_after_run` — Once schedule marks COMPLETED
- `test_schedule_max_runs_stops` — Stops after max_runs
- `test_schedule_pause_resume` — Paused schedules skip execution
- `test_schedule_concurrent_lock` — Multiple schedulers don't duplicate execution
- `test_schedule_lock_expiry` — Crashed scheduler's lock expires

### 9.2 Multi-tenancy Tests

- `test_org_isolation_workflows` — Org A can't see Org B's workflows
- `test_org_isolation_memory` — Org A can't search Org B's memory
- `test_org_quota_rate_limit` — Exceeding rate limit returns 429
- `test_org_quota_workflow_limit` — Exceeding workflow limit returns 429
- `test_org_member_role_permissions` — MEMBER can't delete, ADMIN can
- `test_org_owner_manage_settings` — Only OWNER can update org settings
- `test_cross_namespace_admin_access` — Admin with global scope can access all orgs

### 9.3 Performance Tests

- `test_api_throughput_100rps` — Sustained 100 requests/second for 5 minutes
- `test_workflow_throughput_20wps` — 20 concurrent workflows per second
- `test_p95_latency_under_load` — P95 < 200ms for API reads under load
- `test_memory_stability` — Memory doesn't grow over 30-minute sustained load

## 11. Risks

| Risk | Mitigation |
|------|------------|
| Scheduler single-point-of-failure | Lock-based HA; lease expiry prevents deadlocks |
| Namespace migration complexity | Nullable column initially; backfill and enforce in follow-up |
| Quota overhead | Redis-backed counters; lazy evaluation |
| Load testing reveals bottlenecks | Profile early in week 3; optimize iteratively |

## 12. Rollback

- Scheduling: disable scheduler service; API endpoints return 501
- Multi-tenancy: namespace column exists but ignored; RBAC falls back to global
- Performance: indexes can be dropped; cache is optional
