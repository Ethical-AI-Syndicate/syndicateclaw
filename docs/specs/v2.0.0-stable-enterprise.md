## 1. Problem Statement

By v1.5.0, SyndicateClaw has all the features needed for enterprise agent orchestration: RBAC, LLM integration, multi-agent coordination, scheduling, multi-tenancy, SDK, and visual builder. But it has not been hardened for production: no comprehensive security audit covering v1.3.0–v1.5.0 attack surfaces, incomplete documentation, no tested upgrade path from v1.0, and no confidence in failure modes including the scheduler. A v2.0.0 release must be battle-tested, well-documented, and easy to adopt.

---

## 2. Goals

- **Security hardening**: OWASP-compliant endpoints, secrets handling, input validation — covering all v1.1.0–v1.5.0 attack surfaces
- **Chaos testing**: Verify graceful degradation under DB/Redis/network/scheduler failures
- **Documentation**: Complete API docs, deployment guide, upgrade guide, troubleshooting
- **Release**: Tagged v2.0.0 with changelog, migration guide, and tested rollback procedure

## 4. Security Audit

### 4.1 OWASP Top 10 Coverage

| OWASP Category | SyndicateClaw Mitigation | Audit Task |
|----------------|-------------------------|------------|
| **A01 Broken Access Control** | RBAC enforcement, ownership scoping, namespace isolation | Verify all endpoints check permissions; verify namespace filter on every DB query |
| **A02 Cryptographic Failures** | HMAC signing, Ed25519, TLS | Verify key management, no plaintext secrets, no secrets in audit event details |
| **A03 Injection** | Safe expression evaluator, parameterized queries, Jinja2 sandbox | Verify no `eval()`, SQL injection, XSS in builder UI (see §4.2), second-order template injection via LLM output |
| **A04 Insecure Design** | Fail-closed policy, approval gates | Verify design decisions are documented in ADRs |
| **A05 Security Misconfiguration** | Explicit config, no defaults in prod | Verify production config is restrictive; anonymous fallback disabled |
| **A06 Vulnerable Components** | pip-audit, bandit | Run vulnerability scan on all dependencies (see §4.2 for gate definition) |
| **A07 Auth Failures** | JWT with algorithm allowlist, API key hashing, streaming tokens for SSE/builder | Verify token validation, no JWT in URL for primary auth, brute-force protection |
| **A08 Data Integrity Failures** | HMAC-signed audit events, checkpoint signing | Verify signature verification on reads; verify agent `sender` is server-set |
| **A09 Logging Failures** | Append-only audit log, structured logging | Verify all security events are logged; verify plugin hooks are audited |
| **A10 SSRF** | URL validation, blocklist, DNS resolution | Verify all URL surfaces including `WebhookPlugin` and provider test endpoint |

### 4.2 Specific Audit Tasks

**Input Validation:**
- [ ] All API endpoints validate request bodies against Pydantic schemas
- [ ] JSONB fields in PostgreSQL validated before storage
- [ ] Workflow definition nodes/edges validated on create/update
- [ ] Memory values checked against size/nesting limits
- [ ] Template interpolation uses `jinja2.sandbox.SandboxedEnvironment` with `StrictUndefined`
- [ ] Agent message `content` JSONB validated against size limit (256KB)
- [ ] Workflow builder UI: node names and condition strings are escaped before rendering in react-flow canvas (XSS prevention in builder)

**Secrets Management:**
- [ ] No secrets in logs, error messages, or API responses
- [ ] API keys hashed with SHA-256 before storage
- [ ] JWT signing keys stored in environment variables only; managed via secrets manager in production
- [ ] Provider API keys never written to disk; managed via secrets manager
- [ ] Audit event `details` JSONB: tool arguments are redacted before storage using the same redaction filter applied to workflow state responses
- [ ] Streaming tokens and builder tokens are single-use and expire; never in logs

**Rate Limiting:**
- [ ] All endpoints respect per-actor rate limits
- [ ] Org-level quota enforcement active for all six quota types
- [ ] Health/docs endpoints excluded from rate limiting
- [ ] Rate limit headers present in all responses
- [ ] 429 responses include `Retry-After` header

**CORS:**
- [ ] CORS origins explicitly configured (not wildcard `*` in production)
- [ ] Credentials not allowed with wildcard origins
- [ ] Preflight requests handled correctly
- [ ] Builder endpoint CORS allows only explicitly listed embedding origins

**Dependencies:**
- [ ] `pip-audit` reports zero **critical or high** severity vulnerabilities **with available patches** (see §4.2 note)
- [ ] `bandit` reports zero high-severity issues
- [ ] Server `requirements.txt` / `poetry.lock` pins all dependencies to exact versions
- [ ] SDK `pyproject.toml` uses compatible version ranges (e.g., `httpx>=0.25,<1.0`) — NOT exact pinning, which would break users' environments
- [ ] No unused dependencies

> **`pip-audit` gate note**: "Zero vulnerabilities" is not a meaningful gate — at any given time, CVEs may be published for transitive dependencies with no available fix. The gate is: **zero critical or high severity vulnerabilities where a patched version is available**. CVEs with no available fix are tracked and documented as accepted risks. This is evaluated by `pip-audit --fix --dry-run` and reviewing the output.

### 4.3 Penetration Testing Scenarios

Scenarios cover all attack surfaces introduced from v1.1.0 through v1.5.0.

**Core scenarios (inherited):**
1. **Unauthenticated access**: Attempt to access all endpoints without credentials
2. **Privilege escalation**: MEMBER trying to access ADMIN resources
3. **Cross-org access**: Org A accessing Org B's resources via namespace bypass
4. **SSRF via tools**: Craft URLs that resolve to internal services; verify blocklist coverage
5. **Workflow injection**: Craft workflow definitions that attempt to execute arbitrary code via condition expressions
6. **Memory bomb**: Create deeply nested memory values to exhaust resources
7. **Rate limit bypass**: Attempt to exceed rate limits with concurrent requests
8. **Token replay**: Capture and reuse expired JWTs
9. **Audit tampering**: Attempt to modify audit events via direct DB manipulation; verify HMAC prevents fabrication
10. **Checkpoint injection**: Tamper with checkpoint HMAC signatures; verify replay rejects

**v1.3.0 — Agent Mesh scenarios:**
11. **Agent impersonation via sender spoofing**: Submit `POST /api/v1/messages` with a forged `sender` field; verify server overrides it and logs a warning
12. **BROADCAST amplification**: Actor without `message:broadcast` permission attempts BROADCAST; verify 403
13. **BROADCAST flood**: Actor with permission sends BROADCAST to namespace with >50 subscribers; verify 400
14. **Circular message loop**: Agent A sends to B, B auto-replies to A; verify hop limit terminates the loop before queue exhaustion
15. **Cross-agent prompt injection via messaging**: Craft a message `content` that contains an LLM prompt injection; deliver to a workflow with an `llm` node; verify the message content is treated as untrusted input and does not escape the LLM node's tool-call gating

**v1.4.0 — Enterprise Runtime scenarios:**
16. **Scheduler duplicate execution**: Two scheduler instances targeting the same due schedule simultaneously; verify `SKIP LOCKED` ensures exactly one run is created
17. **Namespace bypass via NULL**: Attempt to insert a resource with `namespace = NULL`; verify DB rejects it
18. **Quota bypass via path variation**: Attempt `POST /api/v1/workflows/` (trailing slash) and other path variants to bypass quota enforcement; verify quota decorator catches all paths
19. **JWT stale permissions**: Change actor's org_role from ADMIN to VIEWER; verify next request (with old JWT) uses fresh DB-resolved permissions, not the stale `org_role` claim

**v1.5.0 — Developer Experience scenarios:**
20. **Plugin state mutation bypass**: Write a plugin that attempts to modify `ctx.state` via `MappingProxyType` bypass techniques; verify `TypeError` is raised and the live state is unchanged
21. **Plugin file-path injection**: Attempt to configure a plugin via file path in `plugins.yaml`; verify `PluginConfigError` at startup
22. **Plugin async task escape**: Write a plugin that spawns `asyncio.create_task()`; verify AST checker rejects it at load time
23. **Builder JWT leakage**: Attempt to load the builder with a primary JWT in the URL query parameter; verify server rejects non-builder-token auth
24. **Builder CSRF**: Attempt to `PUT /api/v1/workflows/{id}` from a cross-origin context without `X-Builder-Token`; verify 403
25. **LocalRuntime policy bypass**: Verify `LocalRuntime` cannot be constructed in an environment where `SYNDICATECLAW_ENVIRONMENT=production`
26. **WebhookPlugin SSRF**: Configure `WebhookPlugin` with a private IP webhook URL; verify `validate_url()` blocks the send

## 5. Chaos Testing

### 5.1 Failure Scenarios

| Scenario | Expected Behavior | Test Method |
|----------|-------------------|-------------|
| **PostgreSQL down** | 503 on health check; existing runs pause; scheduler defers | Stop Postgres during active workflow execution and scheduler polling |
| **Redis down** | Graceful degradation; cache misses hit DB; rate limiting degrades open | Stop Redis during API requests |
| **Provider API timeout** | LLM node retries with backoff; eventually fails with audit event | Mock provider with delayed responses |
| **Provider API 500** | LLM node retries; eventually fails | Mock provider returning 500 |
| **Network partition** | Requests timeout; health check degrades | Block network to Postgres/Redis |
| **Disk full** | Audit events queue in DLQ; error logged; DLQ write also fails gracefully | Fill disk during writes |
| **Memory pressure** | OOM killer triggers; service restarts; in-flight runs resume from checkpoint | Stress memory during workflow execution |
| **Scheduler loses DB mid-execution** | Lock is held; on reconnect, scheduler updates `next_run_at` and clears lock; no duplicate run | Kill DB connection after `SKIP LOCKED` claim but before run creation |
| **Scheduler crashes after lock** | Lock expires after `locked_until`; second scheduler instance picks up the schedule on next poll; exactly one run created | Kill scheduler process after lock claim; verify single run after lock expiry |
| **Two schedulers race on same schedule** | Exactly one scheduler creates a run; the other sees no rows (SKIP LOCKED); verified by run count | Start two scheduler instances; trigger a single due schedule |
| **DLQ write fails (disk full during audit failure)** | Nested DLQ failure is logged as ERROR; process does not crash; primary operation continues | Fill disk; trigger audit write failure simultaneously |

### 5.2 Test Methodology

1. **Inject failures** during sustained load (50 concurrent users, 5 minutes)
2. **Measure**:
   - Recovery time after failure is removed (target: <30 seconds)
   - Error rate during failure
   - Data loss (runs, audit events) — target: zero
   - Degradation: graceful vs. catastrophic
3. **Scheduler-specific measurement**: Count total runs created vs. total scheduled fires; verify 1:1 ratio
4. **Document** all failure modes in `docs/failure-modes.md`

### 5.3 Acceptance Criteria

- [ ] No data loss during any failure scenario
- [ ] Recovery time <30 seconds after failure removal
- [ ] Health endpoint correctly reports degraded state for each failure type
- [ ] Audit events for failures are logged (or queued in DLQ if DB unavailable)
- [ ] No unhandled exceptions crash the process
- [ ] Scheduler produces exactly one run per schedule tick under concurrent HA instances
- [ ] Scheduler lock expires and is recovered after simulated crash

## 6. Performance Benchmarks

### 6.1 Benchmark Scenarios

These targets are validated against the v1.4.0 baselines established in that release. The v2.0.0 targets are set based on measured v1.4.0 baselines plus expected improvements from v1.5.0 optimizations (if any). Where no optimization occurred, the v1.4.0 baseline is the v2.0.0 target.

| Scenario | Target | Basis |
|----------|--------|-------|
| **API read (GET /workflows)** | P95 < 200ms | v1.4.0 baseline target |
| **API write (POST /workflows)** | P95 < 400ms | v1.4.0 baseline target × 2 (conservative for write path) |
| **Simple workflow (3 nodes)** | P95 < 2s | v1.4.0 baseline target |
| **Complex workflow (10 nodes, decisions)** | P95 < 10s | First measurement in v2.0.0 |
| **LLM workflow (2 nodes, 1 LLM call)** | P95 < provider_latency + 2s | Dominated by provider; measure overhead only |
| **Memory search** | P95 < 500ms | First measurement in v2.0.0 |
| **Audit query** | P95 < 300ms | First measurement in v2.0.0 |

> **Note on tightening targets**: The original v2.0.0 spec set a 100ms P95 for `GET /workflows` — half the v1.4.0 200ms target — with no explanation. This revision aligns the targets with v1.4.0 baselines. If v2.0.0 profiling reveals that 100ms is achievable with specific optimizations, those optimizations should be specified and the target updated accordingly.

### 6.2 Benchmark Environment

- **Hardware**: 4 vCPU, 8GB RAM (same as v1.4.0 for comparability)
- **Database**: PostgreSQL 16, dedicated instance
- **Cache**: Redis 7, dedicated instance
- **Workload**: Mixed read/write, 80/20 ratio

### 6.3 Regression Prevention

Weekly benchmark CI job:

```yaml
benchmark:
  stage: test
  script:
    - pip install -e ".[dev]"
    - pytest tests/perf/ --benchmark-only --benchmark-json=benchmark.json
    - python scripts/check_benchmark_regression.py benchmark.json --baseline v1.4.0-baseline.json
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule"'
```

The regression checker compares current results against the stored v1.4.0 baseline and fails if any metric degrades >10%.

### 7.4 Upgrade Guide (v1.0 → v2.0)

```markdown
## Upgrade from v1.0.0 to v2.0.0

### ⚠️ Critical: Back Up Your Database Before Upgrading

**IMPORTANT**: Database migrations from v1.0.0 to v2.0.0 are not reversible without
restoring from backup. Data created after migration (workflow runs, audit events,
agent messages, memory records) will be PERMANENTLY LOST if you restore a pre-upgrade
backup. Take a full database backup immediately before running migrations and verify
the backup is restorable before proceeding.

```bash
# Recommended backup command
pg_dump -Fc syndicateclaw_production > syndicateclaw_v1_backup_$(date +%Y%m%d_%H%M%S).dump

# Verify backup is restorable (in a separate test database)
pg_restore -d syndicateclaw_test syndicateclaw_v1_backup_*.dump
```

### Breaking Changes

1. **RBAC Enforcement** (v1.1.0): RBAC is now enforced by default.
   - Risk: Existing workflows that bypass the old prefix-based checks may receive 403.
   - Action: Review your RBAC role assignments before upgrading. Run in shadow mode
     (`SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false`) for 48 hours post-upgrade and
     monitor logs for unexpected denials before enabling enforcement.

2. **Namespace Scoping** (v1.4.0): All resources now require a namespace.
   - Risk: Resources without a namespace are migrated to namespace `"default"`.
   - Action: Review whether `"default"` is the correct namespace for your existing
     resources. After migration, resources can be moved to other namespaces via the API.

3. **API Key Scope Metadata** (v1.1.0): API keys can carry explicit scope metadata.
   - Action: No immediate action required. Existing keys may remain empty-scope.
   - Current behavior: Request-time authorization remains actor/RBAC-driven; OAuth-style
     per-key scopes are not yet a separate enforcement gate.

4. **Permission vocabulary** (v1.1.0): All permissions use singular resource form
   (`workflow:read`, not `workflows:read`). Any code referencing old permission strings
   must be updated.

5. **JWT claims** (v1.4.0): The `permissions` array is no longer embedded in JWTs.
   Clients that parsed the JWT to extract permissions must use the RBAC API instead.

### Pre-Upgrade Checklist

- [ ] Full database backup taken and verified restorable
- [ ] Staging environment upgraded and tested
- [ ] RBAC role assignments reviewed
- [ ] API key scopes reviewed
- [ ] Plugin configurations reviewed (file-path plugins must be converted to entry points)
- [ ] Rollback procedure tested on staging

### Database Migrations

Run migrations in order:

```bash
alembic upgrade head
```

Migrations applied (in order):
- 006: API key scopes
- 007: Idempotency records
- 008: Streaming tokens
- 009: Agents
- 010: Agent messages
- 011: Topic subscriptions
- 012: Workflow versions table
- 013: Workflow definitions versioning columns
- 014: Organizations
- 015: Organization members
- 016: Workflow schedules
- 017–022: Namespace scoping (one migration per table; each bacfills to 'default')
- 023: Performance indexes
- 024: Plugin event types
- 025: Builder token type

### Rollback Procedure

Rollback after upgrade **will result in permanent data loss** for all data created
after the migration ran. This is not a routine operation — it is a disaster recovery
procedure.

**Before attempting rollback, export critical data:**

```bash
# Export audit events (compliance records) created after upgrade
psql syndicateclaw_production \
  -c "COPY (SELECT * FROM audit_events WHERE created_at > '<UPGRADE_TIMESTAMP>') \
      TO '/tmp/post_upgrade_audit_events.csv' CSV HEADER;"

# Export workflow runs created after upgrade
psql syndicateclaw_production \
  -c "COPY (SELECT * FROM workflow_runs WHERE created_at > '<UPGRADE_TIMESTAMP>') \
      TO '/tmp/post_upgrade_runs.csv' CSV HEADER;"
```

**Rollback steps:**

1. Stop the v2.0.0 service.
2. Export critical post-upgrade data (see above).
3. Restore the pre-upgrade backup:
   ```bash
   pg_restore -d syndicateclaw_production syndicateclaw_v1_backup_*.dump
   ```
4. Start the v1.0.0 service.
5. Verify service health.

**Post-rollback data reconciliation**: Post-upgrade data (runs, audit events, messages,
memory records) is not in the restored backup. If any of this data is required for
compliance or operations, it must be manually reconciled from the CSV exports above.
Contact your compliance officer before performing a rollback if audit event continuity
is required by your organization's policies.
```

### 7.5 Troubleshooting Guide

| Symptom | Cause | Fix |
|---------|-------|-----|
| 403 on all endpoints | RBAC enabled but no permissions configured | Disable RBAC (`SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false`) or configure roles |
| Workflows stuck in RUNNING | PostgreSQL connection pool exhausted | Increase `pool_size`; check slow queries |
| LLM calls fail | Provider API key not set or invalid | Check env vars; verify provider config; use `/api/v1/providers/{name}/test` (admin only) |
| Memory search returns empty | Redis down (cache disabled) | Check Redis; search falls back to DB automatically |
| Audit events missing | Dead letter queue has failed events | Query `dead_letter_records`; run `retry_all()`; check disk space |
| Scheduler not firing | Scheduler service not running or lock stuck | Check `SYNDICATECLAW_SCHEDULER_ENABLED`; check `locked_until` in `workflow_schedules` |
| Duplicate scheduled runs | Two scheduler instances with clock skew | Verify NTP sync; check `locked_until` is set before run creation |
| Plugin hooks not firing | Plugin not in `plugins.yaml` or load error | Check startup logs for `PluginSecurityViolationError` or `PluginConfigError` |
| Builder returns 401 | Builder token expired or not obtained | Re-issue builder token via `POST /api/v1/workflows/{id}/builder-token` |
| Agent messages not delivered | Agent OFFLINE and TTL expired | Check agent heartbeat; verify `agent_heartbeat_timeout_seconds`; check `agent_messages.status` |
