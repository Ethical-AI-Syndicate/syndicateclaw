# v2.0.0 — Stable Enterprise

**Sprint Duration:** 3 weeks
**Goal:** Production hardening, security audit, documentation, and release

---

## 1. Problem Statement

By v1.5.0, SyndicateClaw has all the features needed for enterprise agent orchestration: RBAC, LLM integration, multi-agent coordination, scheduling, multi-tenancy, SDK, and visual builder. But it hasn't been hardened for production: no security audit, incomplete documentation, no upgrade path from v1.0, and no confidence in failure modes. A v2.0.0 release needs to be battle-tested, well-documented, and easy to adopt.

## 2. Goals

- **Security hardening**: OWASP-compliant endpoints, secrets handling, input validation
- **Chaos testing**: Verify graceful degradation under DB/Redis/network failures
- **Documentation**: Complete API docs, deployment guide, upgrade guide, troubleshooting
- **Release**: Tagged v2.0.0 with changelog and migration guide

## 3. Non-Goals

- New features (all feature work done in v1.1–v1.5)
- Kubernetes operator (future)
- Mobile SDK (future)
- Multi-region deployment (future)

## 4. Security Audit

### 4.1 OWASP Top 10 Coverage

| OWASP Category | SyndicateClaw Mitigation | Audit Task |
|----------------|-------------------------|------------|
| **A01 Broken Access Control** | RBAC enforcement, ownership scoping, namespace isolation | Verify all endpoints check permissions |
| **A02 Cryptographic Failures** | HMAC signing, Ed25519, TLS | Verify key management, no plaintext secrets |
| **A03 Injection** | Safe expression evaluator, parameterized queries | Verify no eval(), SQL injection, XSS |
| **A04 Insecure Design** | Fail-closed policy, approval gates | Verify design decisions are documented |
| **A05 Security Misconfiguration** | Explicit config, no defaults in prod | Verify production config is restrictive |
| **A06 Vulnerable Components** | pip-audit, bandit | Run vulnerability scan on all dependencies |
| **A07 Auth Failures** | JWT with algorithm allowlist, API key hashing | Verify token validation, brute-force protection |
| **A08 Data Integrity Failures** | HMAC-signed audit events, checkpoint signing | Verify signature verification on reads |
| **A09 Logging Failures** | Append-only audit log, structured logging | Verify all security events are logged |
| **A10 SSRF** | URL validation, blocklist, DNS resolution | Verify all URL surfaces are protected |

### 4.2 Specific Audit Tasks

**Input Validation:**
- [ ] All API endpoints validate request bodies against Pydantic schemas
- [ ] JSONB fields in PostgreSQL are validated before storage
- [ ] Workflow definition nodes/edges validated on create/update
- [ ] Memory values checked against size/nesting limits
- [ ] Template interpolation is sandboxed (no file access, no imports)

**Secrets Management:**
- [ ] No secrets in logs, error messages, or API responses
- [ ] API keys hashed with SHA-256 before storage
- [ ] JWT signing keys stored in environment variables only
- [ ] Provider API keys never written to disk
- [ ] Audit events redact sensitive fields

**Rate Limiting:**
- [ ] All endpoints respect per-actor rate limits
- [ ] Health/docs endpoints are excluded from rate limiting
- [ ] Rate limit headers present in all responses
- [ ] 429 responses include `Retry-After` header

**CORS:**
- [ ] CORS origins explicitly configured (not wildcard in production)
- [ ] Credentials not allowed with wildcard origins
- [ ] Preflight requests handled correctly

**Dependencies:**
- [ ] `pip-audit` reports zero vulnerabilities
- [ ] `bandit` reports zero high-severity issues
- [ ] All dependencies pinned to exact versions
- [ ] No unused dependencies

### 4.3 Penetration Testing Scenarios

1. **Unauthenticated access**: Attempt to access all endpoints without credentials
2. **Privilege escalation**: MEMBER trying to access ADMIN resources
3. **Cross-org access**: Org A accessing Org B's resources
4. **SSRF via tools**: Craft URLs that resolve to internal services
5. **Workflow injection**: Craft workflow definitions that execute arbitrary code
6. **Memory bomb**: Create deeply nested memory values to exhaust resources
7. **Rate limit bypass**: Attempt to exceed rate limits with concurrent requests
8. **Token replay**: Capture and reuse expired JWTs
9. **Audit tampering**: Attempt to modify audit events
10. **Checkpoint injection**: Tamper with checkpoint HMAC signatures

## 5. Chaos Testing

### 5.1 Failure Scenarios

| Scenario | Expected Behavior | Test Method |
|----------|-------------------|-------------|
| **PostgreSQL down** | 503 on health check; existing runs pause gracefully | Stop Postgres during workflow execution |
| **Redis down** | Graceful degradation; cache misses hit DB | Stop Redis during API requests |
| **Provider API timeout** | LLM node retries with backoff; eventually fails | Mock provider with delayed responses |
| **Provider API 500** | LLM node retries; eventually fails | Mock provider returning 500 |
| **Network partition** | Requests timeout; health check degrades | Block network to Postgres/Redis |
| **Disk full** | Audit events queue; error logged | Fill disk during writes |
| **Memory pressure** | OOM killer; service restarts | Stress memory during workflow execution |

### 5.2 Test Methodology

1. **Inject failures** during sustained load (50 concurrent users, 5 minutes)
2. **Measure**:
   - Recovery time after failure is removed
   - Error rate during failure
   - Data loss (runs, audit events)
   - Degradation graceful vs. catastrophic
3. **Document** failure modes in `docs/failure-modes.md`

### 5.3 Acceptance Criteria

- [ ] No data loss during any failure scenario
- [ ] Recovery time < 30 seconds after failure removal
- [ ] Health endpoint correctly reports degraded state
- [ ] Audit events for failures are logged
- [ ] No unhandled exceptions crash the process

## 6. Performance Benchmarks

### 6.1 Benchmark Scenarios

| Scenario | Target | Measurement |
|----------|--------|-------------|
| **API read (GET /workflows)** | P95 < 100ms | Sustained 100 req/s for 5 minutes |
| **API write (POST /workflows)** | P95 < 200ms | Sustained 50 req/s for 5 minutes |
| **Simple workflow (3 nodes)** | P95 < 1s | 20 concurrent executions |
| **Complex workflow (10 nodes, decisions)** | P95 < 5s | 10 concurrent executions |
| **LLM workflow (2 nodes, 1 LLM call)** | P95 < 10s | 5 concurrent executions |
| **Memory search** | P95 < 500ms | 10K records, full-text search |
| **Audit query** | P95 < 300ms | 100K records, filtered query |

### 6.2 Benchmark Environment

- **Hardware**: 4 vCPU, 8GB RAM (or equivalent cloud instance)
- **Database**: PostgreSQL 16, dedicated instance
- **Cache**: Redis 7, dedicated instance
- **Workload**: Mixed read/write, 80/20 ratio

### 6.3 Regression Prevention

Add benchmark CI job that runs weekly:

```yaml
benchmark:
  stage: test
  script:
    - pip install -e ".[dev]"
    - pytest tests/perf/ --benchmark-only --benchmark-json=benchmark.json
    - python scripts/check_benchmark_regression.py benchmark.json
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule"'
```

The regression checker compares current results against a stored baseline and fails if performance degrades > 10%.

### 7.4 Upgrade Guide (v1.0 → v2.0)

```markdown
## Upgrade from v1.0.0 to v2.0.0

### Breaking Changes

1. **RBAC Enforcement**: RBAC is now enforced by default.
   - Action: Review your RBAC roles/permissions before upgrading.
   - Fallback: Set `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false` to disable.

2. **Namespace Scoping**: All resources now require a namespace.
   - Action: Existing resources are migrated to namespace `default`.
   - Fallback: Set `SYNDICATECLAW_ORGANIZATION_ENABLED=false`.

3. **API Key Scopes**: API keys can now have explicit scopes.
   - Action: Existing keys retain full access (empty scopes = full access).
   - No action required for backward compatibility.

### Database Migrations

Run migrations in order:

```bash
alembic upgrade head
```

Migrations applied:
- 006: API key scopes
- 007: Idempotency records
- 008: Agents
- 009: Agent messages
- 010: Workflow versions
- 011: Organizations
- 012: Organization members
- 013: Workflow schedules
- 014: Namespace scoping
- 015: Performance indexes

### Rollback

To rollback to v1.0.0:
1. Stop the v2.0.0 service
2. Restore database from pre-upgrade backup
3. Start v1.0.0 service

Note: Data created after the upgrade will be lost.
```

### 7.5 Troubleshooting Guide

| Symptom | Cause | Fix |
|---------|-------|-----|
| 403 on all endpoints | RBAC enabled but no permissions configured | Disable RBAC or configure roles |
| Workflows stuck in RUNNING | PostgreSQL connection pool exhausted | Increase `pool_size` |
| LLM calls fail | Provider API key not set | Check env vars, restart service |
| Memory search returns empty | Redis down (cache disabled) | Check Redis; search falls back to DB |
| Audit events missing | Dead letter queue full | Check `dead_letter_records` table; manual resolution |
| Scheduler not firing | Scheduler service not running | Check `SYNDICATECLAW_SCHEDULER_ENABLED` |
