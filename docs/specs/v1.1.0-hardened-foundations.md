## 1. Problem Statement

SyndicateClaw v1.0.0 shipped with RBAC in **shadow mode** — the evaluator runs but does not enforce decisions. This means any authenticated actor can access any resource. Additionally, accumulated lint and type errors create maintenance friction and onboarding confusion. Without integration tests against a real database, we cannot confidently assert that governance features work end-to-end.

---

## 2. Goals

- **RBAC enforced**: Every request passes through the RBAC evaluator; shadow decisions become authoritative
- **Per-API-key scopes**: API keys carry explicit permission scopes beyond just "valid"
- **Zero debt**: `ruff check src tests` and `mypy src` pass cleanly with no pre-existing errors
- **Integration coverage**: Policy, audit, approval, authz, tool executor, and memory modules have ≥80% integration test coverage

---

## 3. Non-Goals

- Multi-agent coordination (v1.3.0)

## 4. RBAC Enforcement

### 4.1 Current State

The RBAC evaluator (`syndicateclaw.authz.evaluator`) runs in shadow mode. It evaluates every request and logs the decision but does not block access. The legacy prefix-based and ownership-based checks remain authoritative.

Control flag: `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED` (currently defaults to `false`).

### 4.2 Design

#### 4.2.1 Enforcement Toggle

Change the default of `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED` to `true`.

> **Important**: The enforcement toggle is evaluated by its value alone — not by environment name inference. Test suites and local development must explicitly set `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false` to disable enforcement. There is no automatic bypass for environments named `test`, `dev`, or `staging`.

When `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=true`:

1. The RBAC middleware intercepts every request.
2. It resolves the actor's principals → roles → permissions.
3. It evaluates the request against the route registry.
4. If the evaluator returns `DENY`, the request is rejected with HTTP 403 before any further processing, including the policy engine.
5. If the evaluator returns `ALLOW`, the request proceeds to the policy engine (see §4.2.4).

When `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false` (development/test mode):

- Legacy prefix-based checks remain in effect.
- A `WARN` log is emitted on every request: `rbac.enforcement_disabled actor=<> path=<>`.
- This mode must not be enabled in production or staging environments. The application startup sequence checks the environment and emits an `ERROR` log (not a warning) if enforcement is disabled and `SYNDICATECLAW_ENVIRONMENT` is not one of `development`, `dev`, `test`, `testing`.

#### 4.2.2 RBAC + Policy Engine Evaluation Order

Both the RBAC middleware and the policy engine gate requests. Their interaction is:

```
Request
  │
  ▼
RBAC Middleware
  ├── DENY  → HTTP 403 immediately (policy engine NOT consulted)
  └── ALLOW
        │
        ▼
      Policy Engine (for tool execution and resource-level rules)
        ├── DENY             → HTTP 403
        ├── REQUIRE_APPROVAL → HTTP 202 / approval flow
        └── ALLOW            → Request proceeds
```

Rules:
- **RBAC DENY always wins.** The policy engine is not consulted when RBAC denies.
- **Both must ALLOW for the request to succeed.** An RBAC ALLOW does not override a policy engine DENY.
- **The RBAC check is route-level.** The policy engine check is resource-level (tool risk, conditions, audit). They are complementary, not redundant.
- **Tool invocations within workflow execution** (e.g., the `llm` node handler, the tool executor called from a workflow run) are subject to the policy engine. The actor for these checks is the workflow engine's service account (`system:engine`). RBAC enforcement on internal engine calls uses the service account's permissions, not the original API caller's permissions.

#### 4.2.3 Route Registry

All route handlers must register their required permission in the RBAC route registry (`syndicateclaw/authz/route_registry.py`).

**Default for unregistered routes: DENY.** Any route not present in the registry will receive HTTP 403 for all actors except `platform_admin`. New routes must be added to the registry before they are deployed.

#### 4.2.4 Shadow Log Retention and Cleanup

Shadow mode decision logs (`rbac.shadow_decision`) must be **retained through the enforcement validation window** (see §11 Rollout Plan). They are the evidence base for confirming that the RBAC evaluator and legacy prefix checks agree before enforcement is declared stable.

Shadow logs are deleted only after:
1. RBAC enforcement has been enabled in production for at least 7 days.
2. Zero shadow-vs-enforcement disagreements have been recorded in that window.
3. The deletion is a deliberate, separately gated migration step — not bundled with the enforcement flip.

### 4.3 Per-API-Key Scopes

#### 4.3.1 Schema Change

```sql
ALTER TABLE api_keys ADD COLUMN scopes TEXT[] DEFAULT '{}';
-- NOTE: An empty scopes array ('{}'`) is treated as "full access" for
-- backward compatibility with v1.0 keys. This is intentional and
-- documented. Do NOT change this default to restrict access without
-- running the v1.1.0 scope migration (see §4.3.2).
```

#### 4.3.2 Scope Resolution and Migration Path

When an API key is verified:

1. Load the key's `scopes` from the database.
2. If `scopes` is `null` or empty (`{}`), the key is considered **unscoped**:
   - The key is granted full access (backward compatibility with v1.0).
   - A `WARN` log is emitted: `api_key.unscoped key_id=<> actor=<>`.
   - An `unscoped` flag is set on the request context for downstream audit.
3. If `scopes` is set and non-empty, each request is checked against the key's scopes.
4. Scope matching uses exact string comparison **only**. The platform may internally match a key's listed scope against a requested permission using fnmatch-style globs (e.g., `admin:*` covers `admin:users`) — but clients **cannot submit glob patterns** in their scope list (see §4.3.3).

**Unscoped key deprecation timeline:**

| Phase | Action |
|-------|--------|
| v1.1.0 release | Unscoped keys continue to work; `WARN` log on each use |
| v1.2.0 release | `GET /api/v1/api-keys` response includes `"scoped": false` field on unscoped keys; deprecation notice in API response headers |
| v1.3.0 release | Unscoped keys require explicit `SYNDICATECLAW_ALLOW_UNSCOPED_KEYS=true` override to function; default changes to deny |

This timeline must be documented in the upgrade guide shipped with v1.1.0.

#### 4.3.3 Scope Validation on Key Creation

`POST /api/v1/api-keys` accepts an optional `scopes` array. Validation rules:

1. **Unknown scopes are rejected**: Each submitted scope must exactly match an entry in the permission registry. Unknown scopes return HTTP 400.
2. **Glob characters are prohibited**: Scope strings containing `*`, `?`, `[`, or `]` are rejected with HTTP 400. Glob expansion is a platform-internal mechanism only — clients must submit explicit permission strings.
3. **Scope count limit**: A key may carry at most **50 scopes**. Requests exceeding this limit return HTTP 400.
4. **Privilege ceiling**: A key cannot be created with scopes exceeding the creating actor's own permissions. An `operator` cannot create a key with `admin:*`. Attempts return HTTP 403.

#### 4.3.4 Scope Enumeration API

```
GET /api/v1/api-keys/scopes
```

**Authentication**: Any valid JWT or API key (no specific scope required). This endpoint lists platform capabilities, not sensitive data.

Returns the list of all valid scopes:

### 5.1 Scope

Execute the plan in `docs/superpowers/plans/2026-03-25-quality-debt-elimination.md`:

### 5.2 Acceptance Criteria

- `ruff check src tests` exits 0
- `mypy src` exits 0
- `pytest tests/ -q` exits 0 (no regressions introduced by type fixes)

### 5.3 CI Gate

Add a new CI job `quality_gate` that fails the pipeline if either check is non-zero:

```yaml
quality_gate:
  stage: test
  script:
    - ruff check src tests
    - mypy src
```

This is the canonical enforcement point for lint and type quality. Do **not** duplicate these checks as pytest test cases — version drift between the test runner environment and the CI linter environment causes false passes. The CI job runs in the same container image as the linter installation.

## 6. Integration Test Suite

### 6.1 Test Infrastructure

- Use `pytest-asyncio` with `asyncio_mode = "auto"`.
- Tests run against a real PostgreSQL database (via Docker or CI service container).
- Use `factory-boy` for test data generation.
- Use `httpx.AsyncClient` for API endpoint testing.
- All integration tests must set `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false` in the test environment via the shared conftest, or explicitly via `pytest` env fixtures. They must not rely on the test environment name to disable enforcement.

### 6.2 Test Coverage Targets

| Module | Coverage Target | Focus Areas |
|--------|----------------|-------------|
| **Policy Engine** | ≥85% | Rule matching, condition evaluation, fail-closed behavior, CRUD with audit |
| **Audit Service** | ≥85% | Append-only constraint, HMAC signing, dead letter queue, query filters |
| **Approval Service** | ≥80% | Lifecycle transitions, assignee enforcement, self-approval prevention, expiration |
| **RBAC/Authz** | ≥80% | Principal resolution, permission evaluation, route registry enforcement, RBAC+policy interaction |

## 10. Testing Strategy

### 10.1 RBAC Enforcement Tests

- `test_rbac_enforces_valid_permission` — Actor with correct permission can access resource
- `test_rbac_denies_insufficient_permission` — Actor without permission receives 403
- `test_rbac_admin_wildcard` — Actor with `admin:*` scope can access all registered routes
- `test_rbac_deny_beats_role_grant` — A deny assignment blocks an actor even if their role would allow
- `test_rbac_enforcement_off_requires_explicit_flag` — Enforcement is only disabled when `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false` is set; environment name alone does not disable it
- `test_rbac_warn_logged_when_disabled` — Warning log emitted on every request when enforcement is disabled
- `test_rbac_unknown_route_defaults_deny` — Unregistered routes return 403
- `test_rbac_policy_engine_deny_wins_after_rbac_allow` — RBAC ALLOW + policy DENY → 403
- `test_rbac_policy_engine_not_consulted_after_rbac_deny` — Policy engine is not called when RBAC denies (verify via mock assert_not_called)

### 10.2 API Key Scope Tests

- `test_key_with_scopes_restricts_access` — Key with limited scopes cannot access out-of-scope endpoints
- `test_key_without_scopes_has_full_access` — v1.0 keys (empty scopes) retain backward compatibility
- `test_unscoped_key_emits_warn_log` — Warning log emitted on every use of an unscoped key
- `test_key_scope_internal_glob_matching` — Platform internally matches `admin:*` against `admin:users`
- `test_key_scope_client_glob_rejected` — Client-submitted scope `"workflows:*"` is rejected at creation time (HTTP 400)
- `test_key_invalid_scope_rejected_on_create` — Unknown scopes rejected at creation time
- `test_key_scope_count_limit` — Creating a key with 51 scopes returns HTTP 400
- `test_key_scope_privilege_ceiling` — Operator cannot create key with `admin:*` scope (HTTP 403)
- `test_scope_enumeration_endpoint_authenticated` — Authenticated principal can call `GET /api/v1/api-keys/scopes`
- `test_scope_enumeration_endpoint_unauthenticated` — Unauthenticated request returns 401
- `test_scope_permission_vocabulary_singular` — All scopes in enumeration use singular resource form (`workflow:read`, not `workflows:read`)

### 10.3 Route Registry Tests

- `test_all_registered_routes_have_permissions` — Programmatic check: every route in the FastAPI router is present in the route registry
- `test_unregistered_route_denied` — A synthetic route not in the registry returns 403
- `test_route_registry_permission_strings_valid` — Every permission string in the registry exists in the permission vocabulary

## 13. Rollback

- **RBAC enforcement**: Set `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED=false` — takes effect immediately, no redeploy required if env var is externalized.
- **API key scopes**: `scopes` column can be ignored; empty/null continues to grant full access. No migration required.
- **Quality debt**: `git revert` on the cleanup commits. CI gate removed until re-introduced.
