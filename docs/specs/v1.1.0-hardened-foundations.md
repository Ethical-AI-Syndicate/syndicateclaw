# v1.1.0 — Hardened Foundations

**Sprint Duration:** 3 weeks
**Goal:** Close governance gaps, eliminate code quality debt, build confidence in test coverage

---

## 1. Problem Statement

SyndicateClaw v1.0.0 shipped with RBAC in **shadow mode** — the evaluator runs but doesn't enforce decisions. This means any authenticated actor can access any resource. Additionally, accumulated lint and type errors create maintenance friction and onboarding confusion. Without integration tests against a real database, we can't confidently assert that governance features work end-to-end.

## 2. Goals

- **RBAC enforced**: Every request passes through the RBAC evaluator; shadow decisions become authoritative
- **Per-API-key scopes**: API keys carry explicit permission scopes beyond just "valid"
- **Zero debt**: `ruff check src tests` and `mypy src` pass cleanly with no pre-existing errors
- **Integration coverage**: Policy, audit, approval, and authz modules have ≥80% integration test coverage

## 3. Non-Goals

- Multi-agent coordination (v1.3.0)
- LLM integration (v1.2.0)
- Workflow versioning (v1.3.0)
- SDK or visual builder (v1.5.0)

## 4. RBAC Enforcement

### 4.1 Current State

The RBAC evaluator (`syndicateclaw.authz.evaluator`) runs in shadow mode. It evaluates every request and logs the decision but does not block access. The legacy prefix-based and ownership-based checks remain authoritative.

Control flag: `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED` (currently defaults to `false`).

### 4.2 Design

**4.2.1 Enforcement Toggle**

Change the default of `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED` to `true`. When enabled:

1. The RBAC middleware intercepts every request.
2. It resolves the actor's principals → roles → permissions.
3. It evaluates the request against the route registry.
4. If the evaluator returns `DENY`, the request is rejected with HTTP 403.
5. If the evaluator returns `ALLOW`, the request proceeds.

When disabled (development/test mode only), legacy prefix-based checks remain in effect with a warning log.

**4.2.2 Route Registry Update**

All route handlers must register their required permission in the RBAC route registry:

```python
@router.get("/workflows", tags=["workflows"])
async def list_workflows(
    actor: str = Depends(get_current_actor),
    # ... RBAC checks work via route registry, not inline
):
```

The route registry maps `(method, path_pattern)` → `Permission`:

| Route | Permission |
|-------|-----------|
| `GET /api/v1/workflows` | `workflows:read` |
| `POST /api/v1/workflows` | `workflows:create` |
| `GET /api/v1/workflows/{id}` | `workflows:read` |
| `PUT /api/v1/workflows/{id}` | `workflows:update` |
| `DELETE /api/v1/workflows/{id}` | `workflows:delete` |
| `POST /api/v1/workflows/{id}/runs` | `runs:create` |
| `POST /api/v1/workflows/{id}/runs/{rid}/pause` | `runs:manage` |
| `POST /api/v1/approvals/{id}/approve` | `approvals:write` |
| `POST /api/v1/approvals/{id}/reject` | `approvals:write` |
| `POST /api/v1/policies` | `policies:write` |
| `DELETE /api/v1/policies/{id}` | `policies:write` |
| `POST /api/v1/tools/{name}/execute` | `tools:execute` |
| `GET /api/v1/audit` | `audit:read` |

Full registry in `syndicateclaw/authz/route_registry.py`.

**4.2.3 Shadow Log Cleanup**

Remove all shadow-mode log entries (`rbac.shadow_decision`) once enforcement is live. Keep the evaluation logging for audit, but mark enforcement decisions clearly.

### 4.3 Per-API-Key Scopes

**4.3.1 Schema Change**

Add a `scopes` field to the `api_keys` table:

```sql
ALTER TABLE api_keys ADD COLUMN scopes TEXT[] DEFAULT '{}';
```

Scopes are a list of strings like `["workflows:read", "workflows:write", "admin:*"]`.

**4.3.2 Scope Resolution**

When an API key is verified:

1. Load the key's `scopes` from the database.
2. If `scopes` is empty or `null`, default to full access (backward-compatible with v1.0 keys).
3. If `scopes` is set, each request is checked against the key's scopes.
4. Scope matching uses fnmatch-style globs: `workflows:*` matches `workflows:read`, `workflows:create`, etc.
5. `admin:*` grants all permissions.

**4.3.3 Scope Validation on Creation**

`POST /api/v1/api-keys` accepts an optional `scopes` array. Scopes are validated against the permission registry — unknown scopes are rejected.

**4.3.4 Scope Enumeration API**

```
GET /api/v1/api-keys/scopes
```

Returns the list of all valid scopes (for UI dropdowns, documentation):

```json
{
  "scopes": [
    "workflows:read",
    "workflows:write",
    "runs:create",
    "runs:manage",
    "approvals:write",
    "policies:write",
    "tools:execute",
    "memory:read",
    "memory:write",
    "audit:read",
    "admin:*"
  ]
}
```

## 5. Quality Debt Elimination

### 5.1 Scope

Execute the plan in `docs/superpowers/plans/2026-03-25-quality-debt-elimination.md`:

1. **Ruff lane**: Fix all existing ruff errors (auto-fix where safe, manual for the rest).
2. **Mypy lane**: Fix all existing mypy errors in type-stable order (generics → return types → union narrowing → any/no-return cleanup).
3. **Verification**: Run full pytest suite after each lane to confirm behavior stability.

### 5.2 Acceptance Criteria

- `ruff check src tests` exits 0
- `mypy src` exits 0
- `pytest tests/ -q` exits 0 (no regressions introduced by type fixes)

### 5.3 CI Gate

Add a new CI job `quality_gate` that fails the pipeline if either check is non-zero:

```yaml
quality_gate:
  stage: test
  script:
    - ruff check src tests
    - mypy src
```

This ensures no new debt accumulates.

## 6. Integration Test Suite

### 6.1 Test Infrastructure

- Use `pytest-asyncio` with `asyncio_mode = "auto"`.
- Tests run against a real PostgreSQL database (via Docker or CI service container).
- Use `factory-boy` for test data generation.
- Use `httpx.AsyncClient` for API endpoint testing.

### 6.2 Test Coverage Targets

| Module | Coverage Target | Focus Areas |
|--------|----------------|-------------|
| **Policy Engine** | ≥85% | Rule matching, condition evaluation, fail-closed behavior, CRUD with audit |
| **Audit Service** | ≥85% | Append-only constraint, HMAC signing, dead letter queue, query filters |
| **Approval Service** | ≥80% | Lifecycle transitions, assignee enforcement, self-approval prevention, expiration |
| **RBAC/Authz** | ≥80% | Principal resolution, permission evaluation, route registry enforcement |
| **Tool Executor** | ≥80% | Policy gate, sandbox enforcement, timeout, decision ledger requirement |
| **Memory Service** | ≥75% | CRUD, provenance tracking, access policy, namespace isolation, caching |

### 6.3 Test Fixtures

Create shared fixtures in `tests/conftest.py`:

- `db_engine` — PostgreSQL engine (connects to test DB)
- `db_session` — Session-scoped transaction (rolls back after each test)
- `policy_engine` — Pre-configured with test rules
- `audit_service` — Connected to test DB
- `approval_service` — With notification callback disabled
- `test_actor` — Standard authenticated actor
- `admin_actor` — Admin-scoped actor

### 6.4 CI Integration

The integration tests require a running PostgreSQL. In CI, use a service container:

```yaml
integration_tests:
  stage: test
  services:
    - postgres:16
  variables:
    POSTGRES_USER: syndicateclaw
    POSTGRES_PASSWORD: syndicateclaw
    POSTGRES_DB: syndicateclaw_test
    SYNDICATECLAW_DATABASE_URL: postgresql+asyncpg://syndicateclaw:syndicateclaw@postgres:5432/syndicateclaw_test
  script:
    - pytest tests/ -m integration -q --tb=line
```

## 10. Testing Strategy

### 10.1 RBAC Enforcement Tests

- `test_rbac_enforces_valid_permission` — Actor with correct permission can access resource
- `test_rbac_denies_insufficient_permission` — Actor without permission receives 403
- `test_rbac_admin_bypass` — Admin role can access all resources
- `test_rbac_disabled_in_dev` — RBAC disabled when env is development
- `test_rbac_disabled_in_test` — RBAC disabled when env is test
- `test_rbac_unknown_route_defaults_deny` — Unregistered routes are denied

### 10.2 API Key Scope Tests

- `test_key_with_scopes_restricts_access` — Key with limited scopes can't access all endpoints
- `test_key_without_scopes_has_full_access` — v1.0 keys retain backward compatibility
- `test_key_scope_glob_matching` — `workflows:*` matches `workflows:read`
- `test_key_invalid_scope_rejected_on_create` — Unknown scopes rejected at creation time
- `test_scope_enumeration_endpoint` — Lists all valid scopes

### 10.3 Quality Debt Tests

- `test_ruff_clean` — Assert ruff exits 0 on the entire codebase
- `test_mypy_clean` — Assert mypy exits 0 on src/

## 11. Rollout Plan

2. **Week 2**: Implement RBAC enforcement + API key scopes. Unit tests pass.
3. **Week 3**: Integration tests. Run against staging. Validate shadow vs. enforcement parity. Release v1.1.0.

## 12. Risks

| Risk | Mitigation |
|------|------------|
| RBAC enforcement breaks existing workflows | Shadow mode runs in parallel for 1 week post-release; can disable via flag |
| Quality debt fixes introduce regressions | Run full pytest after each fix; CI gates prevent merge if tests fail |
| API key scope changes are backward-incompatible | Empty scopes default to full access; no migration needed |

## 13. Rollback

- Revert `SYNDICATECLAW_RBAC_ENFORCEMENT_ENABLED` to `false`
- API key scopes: remove column or ignore (backward-compatible)
- Quality debt: `git revert` on the cleanup commits
