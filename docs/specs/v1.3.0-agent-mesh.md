## 1. Problem Statement

SyndicateClaw executes single-workflow graphs. There is no way for agents to communicate, delegate tasks, or coordinate across workflows. Real-world agent systems require multiple specialized agents working together — a triage agent that delegates to a research agent, which reports to an approval agent. Without a messaging protocol, users must embed all logic in a single monolithic workflow.

---

## 2. Goals

- **Agent registry**: Agents can register, declare capabilities, and be discovered
- **Messaging protocol**: Agents send and receive messages (direct, broadcast, topic) with loop protection and identity enforcement
- **Workflow integration**: Workflow nodes can send messages to other agents
- **Workflow versioning**: Workflows can be versioned and rolled back via a proper relational schema

---

## 3. Non-Goals

- Agent-to-agent discovery via gossip protocol (future)
- Agent resource limits per tenant (v1.4.0)
- Agent marketplace or public registry (future)

## 4. Agent Registry

### 4.1 Agent Model

```python
class Agent(BaseModel):
    id: str                    # ULID — canonical identity for routing
    name: str                  # Human-readable, unique within namespace
    description: str | None
    namespace: str             # Organizational namespace
    capabilities: list[str]    # e.g., ["research", "summarization", "code-review"]
    metadata: dict[str, Any]   # Arbitrary key-value data (max 64 keys, max 1KB total)
    status: AgentStatus        # ONLINE | OFFLINE | DRAINING
    registered_by: str         # Actor who registered this agent (set server-side, immutable)
    heartbeat_at: datetime     # Last heartbeat timestamp
    created_at: datetime
    updated_at: datetime
```

`registered_by` is set server-side from the authenticated actor at registration time and cannot be modified by the client. It is the ownership field used for all authorization checks on agent mutations.

### 4.2 RBAC Permissions for Agent Resources

All new agent endpoints must be added to the v1.1.0 route registry before deployment.

| Endpoint | Method | Required Permission |
|----------|--------|---------------------|
| `POST /api/v1/agents` | POST | `agent:register` |
| `GET /api/v1/agents` | GET | `agent:read` |
| `GET /api/v1/agents/{id}` | GET | `agent:read` |
| `PUT /api/v1/agents/{id}` | PUT | `agent:manage` (own agent only, see §4.6) |
| `DELETE /api/v1/agents/{id}` | DELETE | `agent:manage` (own agent only, see §4.6) |
| `POST /api/v1/agents/{id}/heartbeat` | POST | `agent:heartbeat` (own agent only, see §4.6) |

New permissions to add to the permission registry and scope enumeration endpoint:

| Permission | Min Role | Description |
|------------|----------|-------------|
| `agent:read` | `viewer` | View agent registry |
| `agent:register` | `operator` | Register a new agent |
| `agent:manage` | `operator` | Update or deregister own agent |
| `agent:heartbeat` | `operator` | Send heartbeat for own agent |
| `agent:admin` | `admin` | Manage any agent in namespace |
| `message:send` | `operator` | Send direct or topic messages |
| `message:broadcast` | `admin` | Send broadcast messages (elevated — see §5.2) |
| `message:read` | `viewer` | Read messages addressed to self |
| `message:ack` | `operator` | Acknowledge message delivery |

### 4.3 API Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `POST /api/v1/agents` | POST | Register an agent |
| `GET /api/v1/agents` | GET | List agents (filtered by namespace, capability, status) |
| `GET /api/v1/agents/{id}` | GET | Get agent details |
| `PUT /api/v1/agents/{id}` | PUT | Update agent (name, capabilities, metadata) |
| `DELETE /api/v1/agents/{id}` | DELETE | Deregister agent |
| `POST /api/v1/agents/{id}/heartbeat` | POST | Agent heartbeat (updates status to ONLINE) |

### 4.4 Agent Discovery

Agents can discover each other by:

1. **By name**: `GET /api/v1/agents?name=triage-agent`
2. **By capability**: `GET /api/v1/agents?capability=research`
3. **By namespace**: `GET /api/v1/agents?namespace=production`
4. **By status**: `GET /api/v1/agents?status=online`

Discovery returns agent `id` and `name`. All message routing should prefer `id` over `name` for durability (see §5.7).

### 4.5 Heartbeat & Status

- Agents send periodic heartbeats via `POST /api/v1/agents/{id}/heartbeat`.
- If no heartbeat received within `SYNDICATECLAW_AGENT_HEARTBEAT_TIMEOUT_SECONDS` (default: 60), status transitions to `OFFLINE`.
- `OFFLINE` agents receive messages but delivery is deferred (messages remain `PENDING` in the queue).
- Messages to `DRAINING` agents are routed to the next `ONLINE` agent with matching capabilities.
- **Full-drain fallback**: If all agents matching the capability are `DRAINING` or `OFFLINE`, message routing behavior is determined by the sender's `fallback_strategy` (§6.4). There is no silent drop — the message enters the queue with the appropriate status and an audit event is emitted.

### 4.6 Ownership Enforcement

All agent mutation endpoints (`PUT`, `DELETE`, heartbeat) enforce ownership:

- The requesting actor must match `agent.registered_by`, OR
- The requesting actor must have `agent:admin` permission.

Actors that fail this check receive HTTP 403. This is enforced at the handler level, in addition to the RBAC route registry check.

## 5. Messaging Protocol

### 5.1 Message Model

```python
class AgentMessage(BaseModel):
    id: str                          # ULID
    conversation_id: str             # Groups related messages
    sender: str                      # Set SERVER-SIDE from authenticated actor. Never client-settable.
    recipient: str | None            # Direct recipient — agent ID (preferred) or agent name
    topic: str | None                # Topic-based routing (requires subscription)
    message_type: MessageType        # REQUEST | RESPONSE | BROADCAST | RELAY
    content: dict[str, Any]          # Message payload (max 256KB)
    metadata: dict[str, Any]         # Arbitrary metadata (max 4KB)
    priority: MessagePriority        # LOW | NORMAL | HIGH | CRITICAL
    status: MessageStatus            # PENDING | DELIVERED | ACKED | FAILED | HOP_LIMIT_EXCEEDED
    ttl_seconds: int                 # Time-to-live (default 3600, max 86400)
    hop_count: int                   # Number of times this message has been routed (default 0)
    parent_message_id: str | None    # For RESPONSE and RELAY types; links to originating message
    created_at: datetime
    delivered_at: datetime | None
    acked_at: datetime | None
```

**`sender` enforcement**: The `POST /api/v1/messages` request body must not include a `sender` field. The API handler sets `sender` server-side to the authenticated actor's identity (agent name if the actor is a registered agent, otherwise the actor string). Any client-submitted `sender` value is silently overridden and a `WARN` log is emitted: `message.sender_override actor=<> submitted_sender=<>`.

### 5.2 Message Types and Authorization

| Type | Description | Recipient | Required Permission |
|------|-------------|-----------|---------------------|
| `REQUEST` | Ask another agent to do something | Required | `message:send` |
| `RESPONSE` | Reply to a previous request | Required | `message:send` |
| `BROADCAST` | Send to all subscribed agents in a namespace | Namespace-scoped | `message:broadcast` |
| `RELAY` | Forward a message with modifications | Required | `message:send` |

**BROADCAST constraints:**
- Requires `message:broadcast` permission (elevated beyond basic `message:send`).
- Delivery is restricted to agents that have explicitly subscribed to `BROADCAST` for the namespace (see §5.5 Topic and Broadcast Subscriptions).
- Maximum recipients per BROADCAST: 50. Broadcasts to namespaces with more than 50 subscribed agents return HTTP 400 with a prompt to use topic-based routing instead.
- BROADCAST content size limit: 64KB (stricter than direct messages).
- Broadcast rate limit: max 10 broadcasts per minute per actor.

### 5.3 Message Routing

```python
class MessageRouter:
    MAX_HOPS: int = 10  # Configurable via SYNDICATECLAW_MESSAGE_MAX_HOPS

    async def route(self, message: AgentMessage) -> DeliveryResult:
        # Enforce hop limit before routing
        if message.hop_count >= self.MAX_HOPS:
            await self._mark_hop_limit_exceeded(message)
            raise HopLimitExceededError(message.id, message.hop_count)

        if message.recipient:
            return await self._deliver_direct(message)
        elif message.topic:
            return await self._deliver_topic(message)
        elif message.message_type == "BROADCAST":
            return await self._deliver_broadcast(message)
        else:
            raise ValueError("Message must have recipient, topic, or be a broadcast")

    async def _relay(self, message: AgentMessage, new_recipient: str) -> AgentMessage:
        """Create a relay with incremented hop_count."""
        return message.copy(update={
            "id": new_ulid(),
            "recipient": new_recipient,
            "hop_count": message.hop_count + 1,
            "parent_message_id": message.id,
        })
```

### 5.4 Loop Protection

Every message carries `hop_count: int` (default 0). When `MessageRouter` relays or forwards a message, it increments `hop_count` on the new message. When `hop_count >= MAX_HOPS` (default 10, configurable via `SYNDICATECLAW_MESSAGE_MAX_HOPS`):

1. The message is marked `HOP_LIMIT_EXCEEDED`.
2. A `message.hop_limit_exceeded` audit event is emitted with the full conversation chain.
3. The message is sent to the dead letter queue.
4. The originating workflow node (if any) fails with `MESSAGE_HOP_LIMIT_EXCEEDED`.

### 5.5 Topic and Broadcast Subscriptions

Topic-based routing requires explicit subscription. Agents subscribe to topics and broadcast namespaces via:

| Endpoint | Method | Description | Permission |
|----------|--------|-------------|------------|
| `POST /api/v1/topics/{topic}/subscribe` | POST | Subscribe own agent to a topic | `agent:manage` |
| `DELETE /api/v1/topics/{topic}/subscribe` | DELETE | Unsubscribe own agent from a topic | `agent:manage` |
| `GET /api/v1/topics` | GET | List topics with subscriber counts | `agent:read` |

**Schema:**

```sql
CREATE TABLE topic_subscriptions (
    id TEXT PRIMARY KEY,
    agent_id TEXT NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
    topic TEXT NOT NULL,
    namespace TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (agent_id, topic)
);
CREATE INDEX idx_topic_subscriptions_topic ON topic_subscriptions(topic, namespace);
```

An agent that is not subscribed to a topic will not receive messages sent to that topic. An agent that is not subscribed to the broadcast channel (subscribed via the special topic `"__broadcast__"`) will not receive BROADCAST messages.

### 5.6 Delivery Guarantees

- **At-least-once**: Messages are retried on delivery failure (exponential backoff, max 5 attempts).
- **Ack on consume**: Recipients acknowledge messages via `POST /api/v1/messages/{id}/ack`.
- **Dead letter**: Messages that fail all retries, exceed hop limit, or are routing errors go to the dead letter queue. Audit event: `message.dead_letter`. DLQ entries include a `source_type: "agent_message"` discriminator to distinguish from audit event DLQ entries.
- **TTL**: Messages that expire (no delivery within `ttl_seconds`) are marked `FAILED` and audited.

### 5.7 Agent Name vs. ID Routing

The `recipient` field accepts either an agent ID (ULID) or an agent name. **Agent IDs are strongly preferred** for all programmatic routing because:

- Agent names are unique-per-namespace but can be reused if an agent is deregistered and re-registered. A workflow authored with `"recipient": "research-agent"` may silently route to a different agent after a redeploy.
- Agent IDs are stable and immutable.

When `recipient` is a name, the router resolves it to an ID at send time and records the resolved ID in the message for audit purposes. A `WARN` log is emitted: `message.name_routing_used recipient_name=<> resolved_id=<>`. SDK and documentation should recommend ID-based routing.

### 5.8 API Endpoints

| Endpoint | Method | Description | Permission |
|----------|--------|-------------|------------|
| `POST /api/v1/messages` | POST | Send a message | `message:send` (or `message:broadcast` for BROADCAST type) |
| `GET /api/v1/messages` | GET | List messages addressed to authenticated actor | `message:read` |
| `GET /api/v1/messages/{id}` | GET | Get message details (own messages only) | `message:read` |
| `POST /api/v1/messages/{id}/ack` | POST | Acknowledge message delivery | `message:ack` |
| `POST /api/v1/messages/{id}/reply` | POST | Reply to a message (creates RESPONSE) | `message:send` |

### 5.9 Message Queue Schema

```sql
CREATE TABLE agent_messages (
    id TEXT PRIMARY KEY,
    conversation_id TEXT NOT NULL,
    sender TEXT NOT NULL,          -- Set server-side; never client-controlled
    recipient TEXT,
    topic TEXT,
    message_type TEXT NOT NULL,
    content JSONB NOT NULL DEFAULT '{}',
    metadata JSONB NOT NULL DEFAULT '{}',
    priority TEXT NOT NULL DEFAULT 'NORMAL',
    status TEXT NOT NULL DEFAULT 'PENDING',
    ttl_seconds INTEGER NOT NULL DEFAULT 3600,
    hop_count INTEGER NOT NULL DEFAULT 0,
    parent_message_id TEXT,        -- FK to agent_messages(id), nullable

CREATE INDEX idx_messages_sender ON agent_messages(sender);
```

## 6. Workflow Integration

### 6.1 Agent Node Handler

A new node type `AGENT` that sends a message and optionally waits for a response:

```python
{
    "id": "delegate_to_researcher",
    "type": "AGENT",
    "handler": "agent_send",
    "config": {
        "recipient_id": "01ABCDEF...",      # Preferred: stable agent ID
        "recipient_name": "research-agent", # Fallback: resolved at send time (logs WARN)
        "message_type": "REQUEST",
        "content": {
            "action": "research",
            "query": "{{ state.research_query }}",
            "depth": "{{ state.research_depth | default(3) }}"
        },
        "wait_for_response": true,
        "response_timeout_seconds": 300,
        "response_key": "research_results"
    }
}
```

### 6.2 Behavior

- **`wait_for_response: false`**: Send and continue immediately. If a response arrives later, it is stored in workflow state under `response_key`. The workflow does not wait.
- **`wait_for_response: true`**: Workflow status transitions to `WAITING_AGENT_RESPONSE`. Execution resumes when the matching response arrives or when `response_timeout_seconds` is exceeded.

**Concurrency pool interaction**: `WAITING_AGENT_RESPONSE` runs count against the platform's `max_concurrent_runs` limit (the same as `WAITING_APPROVAL` runs). Operators must account for potentially long-waiting agent runs when sizing `SYNDICATECLAW_MAX_CONCURRENT_RUNS`. The `GET /readyz` health endpoint now reports the count of runs in each waiting state.

### 6.3 Response Matching

Responses are matched by `conversation_id`. When a node sends a request, a unique `conversation_id` is generated and stored in the node execution context. Incoming messages with a matching `conversation_id` and `message_type=RESPONSE` are routed to the waiting workflow node.

### 6.4 Capabilities-Based Routing

Instead of hardcoding a recipient, a node can specify required capabilities:

```python
{
    "config": {
        "capabilities": ["research", "web-search"],
        "fallback_strategy": "queue",   # "fail" | "queue" | "broadcast"
        "message_type": "REQUEST",
        "content": {"action": "research", "query": "..."}
    }
}
```

The router selects an `ONLINE` agent with all matching capabilities. Selection is deterministic: when multiple agents match, the agent with the oldest `heartbeat_at` (longest-running) is preferred.

**Fallback strategies when no matching `ONLINE` agent exists:**

| Strategy | Behavior |
|----------|----------|
| `fail` | Node fails immediately with `NO_CAPABLE_AGENT_AVAILABLE` |
| `queue` | Message queued until a matching agent comes online; TTL applies |
| `broadcast` | Message sent to all matching agents (subscribed); first RESPONSE wins; subsequent responses logged and discarded |

**Full-drain fallback**: When all matching agents are `DRAINING` and `fallback_strategy` is not `queue`, the behavior follows the table above as if no ONLINE agent exists. The routing decision is always audited.

## 7. Workflow Versioning

### 7.1 Design

Workflows currently overwrite on update. Versioning adds immutable version history with a proper relational schema:

- **Separate versions table**: Each update creates a row in `workflow_versions`; the `workflow_definitions` row stores only the current state.
- **Version pinning**: Runs are pinned to a specific version at creation time.
- **Rollback**: Restore a previous version as the current definition (creates a new version entry).
- **Diff tracking**: See what changed between versions.

### 7.2 Schema

```sql
-- Add version tracking to workflow_definitions
ALTER TABLE workflow_definitions
    ADD COLUMN current_version INTEGER NOT NULL DEFAULT 1,
    ADD COLUMN updated_by TEXT;

-- Separate versions table — NOT a JSONB column on the parent row
CREATE TABLE workflow_versions (
    id TEXT PRIMARY KEY,
    workflow_id TEXT NOT NULL REFERENCES workflow_definitions(id) ON DELETE CASCADE,
    version INTEGER NOT NULL,
    definition JSONB NOT NULL,
    changed_by TEXT NOT NULL,
    changed_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    comment TEXT,
    UNIQUE (workflow_id, version)
);

CREATE INDEX idx_workflow_versions_workflow ON workflow_versions(workflow_id, version DESC);
```

**Why a separate table (not JSONB column):** A JSONB array column storing all version history on the `workflow_definitions` row:
- Grows unboundedly with no efficient per-version lookup
- Is subject to read-modify-write race conditions under concurrent updates
- Cannot be efficiently indexed or queried by version number
- Approaches PostgreSQL row size limits for frequently-updated workflows

The `workflow_versions` table provides atomic version creation (single `INSERT`), efficient per-version lookup by `(workflow_id, version)`, and independent retention management.

### 7.3 Version Cap

Version history is capped at 100 versions per workflow. When a workflow exceeds 100 versions, the oldest versions (lowest version numbers) are archived to a `workflow_versions_archive` table before deletion. The cap is enforced at write time, not by a background job.

### 7.4 API Endpoints

All versioning endpoints require `workflow:manage` permission.

| Endpoint | Method | Description |
|----------|--------|-------------|
| `GET /api/v1/workflows/{id}/versions` | GET | List all versions (paginated) |
| `GET /api/v1/workflows/{id}/versions/{v}` | GET | Get a specific version |
| `POST /api/v1/workflows/{id}/rollback` | POST | Roll back to a version |
| `GET /api/v1/workflows/{id}/diff` | GET | Diff between two versions (`?from=1&to=2`) |

**Rollback request body:**
```json
{"version": 2, "comment": "Reverting to pre-approval-step version"}
```

Rollback creates a **new version** equal to the content of the target version. It does not destructively modify version history. Example: rolling back to version 2 from version 5 creates version 6 with the same definition as version 2.

### 7.5 Version Pinning on Runs

`POST /api/v1/workflows/{id}/runs` accepts an optional `version` parameter:

```json
{"version": 2, "input_state": {}}
```

If omitted, the latest version is used. The resolved version number is stored on the `WorkflowRun` as `workflow_version` (immutable after run creation). Rollback does not affect in-flight runs — they continue executing the version they were pinned to.

**Rollback and pinned runs**: Existing runs pinned to a now-rolled-back version continue to execute using their pinned version's definition (stored in `workflow_versions`). The rollback only changes which version new runs receive by default. This behavior is explicit and audited.

### 7.6 Diff API

```
GET /api/v1/workflows/{id}/diff?from=1&to=2
```

Returns a structured JSON diff:
```json
{
  "from_version": 1,
  "to_version": 2,
  "nodes_added": [...],
  "nodes_removed": [...],
  "nodes_changed": [{"node_id": "validate", "changes": {...}}],
  "edges_added": [...],
  "edges_removed": [...],
  "metadata_changed": {"description": {"from": "...", "to": "..."}}
}
```

## 14. Rollback

- **Agent registry**: Disable endpoints (return 501); `agents` table persists but unused.
- **Messaging**: Message send endpoint returns 501; queue tables exist but unused.
- **Topic subscriptions**: Table persists but unused.
- **Versioning**: `workflow_versions` table persists; `workflow_definitions.current_version` column persists but ignored; existing pinned runs retain their `workflow_version` reference (safe — version rows still exist).
