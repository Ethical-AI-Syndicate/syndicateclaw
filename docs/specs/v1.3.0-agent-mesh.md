# v1.3.0 — Agent Mesh

**Sprint Duration:** 4 weeks
**Goal:** Enable multi-agent coordination through a messaging protocol and agent registry

---

## 1. Problem Statement

SyndicateClaw executes single-workflow graphs. There's no way for agents to communicate, delegate tasks, or coordinate across workflows. Real-world agent systems require multiple specialized agents working together — a triage agent that delegates to a research agent, which reports to an approval agent. Without a messaging protocol, users must embed all logic in a single monolithic workflow.

## 2. Goals

- **Agent registry**: Agents can register, declare capabilities, and be discovered
- **Messaging protocol**: Agents send and receive messages (direct, broadcast, topic)
- **Workflow integration**: Workflow nodes can send messages to other agents
- **Workflow versioning**: Workflows can be versioned and rolled back

## 3. Non-Goals

- Agent-to-agent discovery via gossip protocol (future)
- Agent resource limits per tenant (v1.4.0)
- Agent marketplace or public registry (future)

## 4. Agent Registry

### 4.1 Agent Model

```python
class Agent(BaseModel):
    id: str                    # ULID
    name: str                  # Unique identifier (e.g., "triage-agent")
    description: str | None
    namespace: str             # Organizational namespace
    capabilities: list[str]    # e.g., ["research", "summarization", "code-review"]
    metadata: dict[str, Any]   # Arbitrary key-value data
    status: AgentStatus        # ONLINE | OFFLINE | DRAINING
    heartbeat_at: datetime     # Last heartbeat timestamp
    created_at: datetime
    updated_at: datetime
```

### 4.2 API Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `POST /api/v1/agents` | POST | Register an agent |
| `GET /api/v1/agents` | GET | List agents (filtered by namespace, capability, status) |
| `GET /api/v1/agents/{id}` | GET | Get agent details |
| `PUT /api/v1/agents/{id}` | PUT | Update agent (name, capabilities, metadata) |
| `DELETE /api/v1/agents/{id}` | DELETE | Deregister agent |
| `POST /api/v1/agents/{id}/heartbeat` | POST | Agent heartbeat (updates status to ONLINE) |

### 4.3 Agent Discovery

Agents can discover each other by:

1. **By name**: `GET /api/v1/agents?name=triage-agent`
2. **By capability**: `GET /api/v1/agents?capability=research`
3. **By namespace**: `GET /api/v1/agents?namespace=production`
4. **By status**: `GET /api/v1/agents?status=online`

### 4.4 Heartbeat & Status

- Agents send periodic heartbeats (`POST /agents/{id}/heartbeat`).
- If no heartbeat received within `agent_heartbeat_timeout_seconds` (default: 60), status transitions to `OFFLINE`.
- `OFFLINE` agents receive messages but they're queued (not delivered immediately).
- Messages to `DRAINING` agents are routed to the next available agent with matching capabilities.

## 5. Messaging Protocol

### 5.1 Message Model

```python
class AgentMessage(BaseModel):
    id: str                          # ULID
    conversation_id: str             # Groups related messages
    sender: str                      # Agent name or actor
    recipient: str | None            # Direct recipient (agent name)
    topic: str | None                # Topic-based routing
    message_type: MessageType        # REQUEST | RESPONSE | BROADCAST | RELAY
    content: dict[str, Any]          # Message payload
    metadata: dict[str, Any]         # Arbitrary metadata
    priority: MessagePriority        # LOW | NORMAL | HIGH | CRITICAL
    status: MessageStatus            # PENDING | DELIVERED | ACKED | FAILED
    ttl_seconds: int                 # Time-to-live for queuing
    created_at: datetime
    delivered_at: datetime | None
    acked_at: datetime | None
```

### 5.2 Message Types

| Type | Description | Recipient |
|------|-------------|-----------|
| `REQUEST` | Ask another agent to do something | Required |
| `RESPONSE` | Reply to a previous request | Required |
| `BROADCAST` | Send to all agents in a namespace | Namespace-scoped |
| `RELAY` | Forward a message with modifications | Required |

### 5.3 Message Routing

```python
class MessageRouter:
    async def route(self, message: AgentMessage) -> DeliveryResult:
        if message.recipient:
            # Direct routing
            return await self._deliver_direct(message)
        elif message.topic:
            # Topic routing — deliver to all subscribers
            return await self._deliver_topic(message)
        elif message.message_type == "BROADCAST":
            # Broadcast to namespace
            return await self._deliver_broadcast(message)
        else:
            raise ValueError("Message must have recipient, topic, or be a broadcast")
```

### 5.4 Delivery Guarantees

- **At-least-once**: Messages are retried on delivery failure (exponential backoff, max 5 attempts).
- **Ack on consume**: Recipients acknowledge messages by calling `POST /api/v1/messages/{id}/ack`.
- **Dead letter**: Messages that fail all retries go to the dead letter queue (audit event: `message.dead_letter`).
- **TTL**: Messages that expire (no delivery within `ttl_seconds`) are marked `FAILED` and audited.

### 5.5 API Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `POST /api/v1/messages` | POST | Send a message |
| `GET /api/v1/messages` | GET | List messages (filter by recipient, sender, topic, status) |
| `GET /api/v1/messages/{id}` | GET | Get message details |
| `POST /api/v1/messages/{id}/ack` | POST | Acknowledge message delivery |
| `POST /api/v1/messages/{id}/reply` | POST | Reply to a message (creates response) |

### 5.6 Message Queue

Messages are stored in a PostgreSQL-backed queue:

```sql
CREATE TABLE agent_messages (
    id TEXT PRIMARY KEY,
    conversation_id TEXT NOT NULL,
    sender TEXT NOT NULL,
    recipient TEXT,
    topic TEXT,
    message_type TEXT NOT NULL,
    content JSONB NOT NULL DEFAULT '{}',
    metadata JSONB NOT NULL DEFAULT '{}',
    priority TEXT NOT NULL DEFAULT 'NORMAL',
    status TEXT NOT NULL DEFAULT 'PENDING',
    ttl_seconds INTEGER NOT NULL DEFAULT 3600,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    delivered_at TIMESTAMPTZ,
    acked_at TIMESTAMPTZ,
    expires_at TIMESTAMPTZ GENERATED ALWAYS AS (created_at + ttl_seconds * INTERVAL '1 second') STORED
);

CREATE INDEX idx_messages_recipient_status ON agent_messages(recipient, status);
CREATE INDEX idx_messages_topic_status ON agent_messages(topic, status);
CREATE INDEX idx_messages_conversation ON agent_messages(conversation_id);
CREATE INDEX idx_messages_expires ON agent_messages(expires_at) WHERE status = 'PENDING';
```

## 6. Workflow Integration

### 6.1 Agent Node Handler

A new node type `AGENT` that sends a message and optionally waits for a response:

```python
{
    "id": "delegate_to_researcher",
    "type": "AGENT",
    "handler": "agent_send",
    "config": {
        "recipient": "research-agent",
        "message_type": "REQUEST",
        "content": {
            "action": "research",
            "query": "{{ state.research_query }}",
            "depth": "{{ state.research_depth | default(3) }}"
        },
        "wait_for_response": true,
        "response_timeout_seconds": 300,
        "response_key": "research_results"
    }
}
```

### 6.2 Behavior

- **`wait_for_response: false`**: Send and continue immediately. Response stored asynchronously (if received) in workflow state.
- **`wait_for_response: true`**: Block execution until response arrives or timeout. Status transitions to `WAITING_AGENT_RESPONSE`. On timeout, node fails with `WAITING_AGENT_RESPONSE_TIMEOUT`.

### 6.3 Response Matching

Responses are matched by `conversation_id`. When a node sends a request, it generates a `conversation_id`. The response handler checks the incoming response's `conversation_id` and routes it to the correct workflow state.

### 6.4 Capabilities-Based Routing

Instead of hardcoding a recipient, the node can specify capabilities:

```python
{
    "config": {
        "capabilities": ["research", "web-search"],
        "fallback_strategy": "fail",  # or "queue" or "broadcast"
        "message_type": "REQUEST",
        "content": {"action": "research", "query": "..."}
    }
}
```

The router selects an agent with matching capabilities that's `ONLINE`. If no agent matches, behavior depends on `fallback_strategy`:
- `fail`: Node fails immediately
- `queue`: Message queued until a matching agent comes online
- `broadcast`: Message sent to all matching agents (first response wins)

## 7. Workflow Versioning

### 7.1 Design

Workflows currently overwrite on update. Versioning adds:

- **Immutable versions**: Each update creates a new version.
- **Version pinning**: Runs are pinned to a specific version.
- **Rollback**: Restore a previous version as the current definition.
- **Diff tracking**: See what changed between versions.

### 7.2 Schema

Add `version` column to `workflow_definitions`:

```sql
ALTER TABLE workflow_definitions ADD COLUMN version INTEGER NOT NULL DEFAULT 1;
ALTER TABLE workflow_definitions ADD COLUMN versions JSONB NOT NULL DEFAULT '[]';
ALTER TABLE workflow_definitions ADD COLUMN is_latest BOOLEAN NOT NULL DEFAULT true;
```

`versions` stores the history of definitions:

```json
[
  {"version": 1, "definition": {...}, "changed_by": "actor", "changed_at": "...", "comment": "Initial version"},
  {"version": 2, "definition": {...}, "changed_by": "actor", "changed_at": "...", "comment": "Added approval step"}
]
```

### 7.3 API Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `GET /api/v1/workflows/{id}/versions` | GET | List all versions |
| `GET /api/v1/workflows/{id}/versions/{v}` | GET | Get a specific version |
| `POST /api/v1/workflows/{id}/rollback` | POST | Rollback to a version |

### 7.4 Version Pinning on Runs

`POST /api/v1/workflows/{id}/runs` accepts optional `version` parameter:

```json
{"version": 2}
```

If omitted, uses the latest version. If specified, uses that exact version (even if newer versions exist).

### 7.5 Diff API

```
GET /api/v1/workflows/{id}/diff?from=1&to=2
```

Returns a JSON diff between two versions (node additions/removals, edge changes, config changes).

## 12. Risks

| Risk | Mitigation |
|------|------------|
| Message queue PostgreSQL performance | Index on (recipient, status); partition if volume grows |
| Agent name conflicts | Namespace-scoped uniqueness; migration to UUID if needed |
| Circular message loops | TTL + max hops in message metadata; audit trail tracks chains |
| Versioning storage growth | Cap version history (e.g., keep last 50 versions; archive older) |

## 13. Rollback

- Agent registry: disable endpoints (return 501)
- Messaging: message send endpoint returns error; queue tables exist but unused
- Versioning: `is_latest` continues working; `version` column ignored
- Agent node handler: falls back to `ACTION` type (no-op)
