## 1. Problem Statement

SyndicateClaw's `llm` node handler is a placeholder — it annotates state but does not call any model. This makes the platform unusable for LLM-powered workflows, which is the primary use case for agent orchestration. Without native LLM support, users must build custom tool handlers for every model invocation, losing the benefits of the platform's audit, policy, and retry infrastructure.

---

## 2. Goals

- **Native LLM invocation**: `llm` node handlers call real models via a provider abstraction
- **Provider abstraction**: Support OpenAI, Anthropic, Azure, and Ollama through a unified interface
- **Streaming**: LLM responses can be streamed to clients via SSE
- **Observability**: LLM-specific metrics and tracing for model calls
- **Idempotency**: LLM calls are deduplicated by request hash within a run
- **Security**: LLM-originated tool calls are gated by RBAC and policy engine; prompt injection is mitigated

---

## 3. Non-Goals

- Multi-agent messaging (v1.3.0)
- Visual workflow builder (v1.5.0)
- Fine-tuning or model management
- Cost tracking per-call (deferred to enterprise tier)
- Embedding support in this release (the `embed()` method is defined in the adapter protocol for future use; no `llm` node config exposes it in v1.2.0)

## 4. Provider Abstraction Layer

### 4.1 Design

A `ProviderService` mediates all model calls:

```
Workflow Engine → llm node handler → ProviderService → ProviderAdapter → Model API
```

**ProviderAdapter protocol:**

```python
class ProviderAdapter(Protocol):
    async def complete(
        self,
        messages: list[Message],
        model: str,
        tools: list[Tool] | None = None,
        stream: bool = False,
        **kwargs: Any,
    ) -> CompletionResponse | AsyncIterator[StreamChunk]:
        ...

    async def embed(
        self,
        texts: list[str],
        model: str,
    ) -> list[list[float]]:
        ...
        # NOTE: embed() is defined for future use. No llm node config
        # exposes embedding in v1.2.0. Adapters must implement the method
        # (raise NotImplementedError is acceptable for v1.2.0).
```

**Supported providers:**

| Provider | Env Var | Models | Notes |
|----------|---------|--------|-------|
| OpenAI | `OPENAI_API_KEY` | gpt-4o, gpt-4o-mini, o1 | Via openai-python |
| Anthropic | `ANTHROPIC_API_KEY` | claude-sonnet-4-20250514, claude-haiku-4-20250414 | Via anthropic-python |
| Azure | `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_ENDPOINT` | Azure-hosted models | Via openai-python |
| Ollama | `OLLAMA_HOST` | Local models | Via ollama-python |

### 4.2 Provider Config

`providers.yaml` (loaded at startup). Environment variable substitution uses `${VAR_NAME}` syntax, resolved by a dedicated interpolation function at load time (`syndicateclaw.providers.config.interpolate_env_vars`). The interpolation function:

- Uses `os.environ.get(var_name)` for each `${...}` reference.
- Raises `ConfigurationError` at **startup** if a configured provider references an env var that is missing or empty. Provider initialization failure is fatal — the application does not start with a broken provider config.
- Supports `${VAR:-default}` syntax for optional vars with fallbacks (e.g., `${OLLAMA_HOST:-http://localhost:11434}`).

> **Security note**: Provider API keys must be managed via a secrets manager (Vault, AWS Secrets Manager, GCP Secret Manager, etc.) in production — not as plain environment variables on the host. The `${ENV_VAR}` pattern is a reference to the injected secret, not an instruction to store keys in `.env` files.

```yaml
providers:
  anthropic:
    type: anthropic
    api_key: ${ANTHROPIC_API_KEY}
    models:
      - claude-sonnet-4-20250514
      - claude-haiku-4-20250414
  openai:
    type: openai
    api_key: ${OPENAI_API_KEY}
    models:
      - gpt-4o
      - gpt-4o-mini
  azure:
    type: azure
    api_key: ${AZURE_OPENAI_API_KEY}
    endpoint: ${AZURE_OPENAI_ENDPOINT}
    models: []  # Populated from Azure deployment discovery
  ollama:
    type: ollama
    host: ${OLLAMA_HOST:-http://localhost:11434}
    models:
      - llama3
      - codellama

routing:
  default_provider: openai
  rules:
    # Rules are evaluated in order; FIRST MATCH wins.
    # More-specific rules must come before less-specific rules.
    - match: "claude*"
      provider: anthropic
    - match: "llama*"
      provider: ollama
    - match: "codellama*"
      provider: ollama
    - match: "*"
      provider: openai
```

**Routing rule evaluation**: Rules are evaluated top-to-bottom; the first matching rule wins. The `match` field uses fnmatch-style glob matching against the model name only (not the `provider:model` path). More-specific rules must appear before less-specific rules in the YAML. A catch-all `"*"` rule at the end routes unmatched models to the default provider. If no rule matches and no default is configured, `ProviderRegistry.get_adapter()` raises `UnroutableModelError`.

### 4.3 Provider Registry

```python
class ProviderRegistry:
    def __init__(self, config: ProviderConfig):
        self._adapters: dict[str, ProviderAdapter] = {}
        self._routing_rules: list[RoutingRule] = []

    def get_adapter(self, model: str) -> ProviderAdapter:
        """Route model name to appropriate provider adapter.
        Rules evaluated top-to-bottom; first match wins.
        Raises UnroutableModelError if no rule matches.
        """
        ...

    def list_models(self) -> list[ModelInfo]:
        """Return all available models across providers."""
        ...
```

## 5. LLM Node Handler

### 5.1 Node Config

```python
{
    "id": "summarize",
    "type": "ACTION",
    "handler": "llm",
    "config": {
        "provider": "openai",             # Optional; auto-routes if omitted
        "model": "gpt-4o",
        "messages": [
            {"role": "system", "content": "You are a helpful assistant."},
            {"role": "user", "content": "{{ state.input_text }}"}
        ],
        "temperature": 0.7,
        "max_tokens": 1024,
        "allow_tool_calls": false,        # REQUIRED opt-in to enable LLM tool-calling (default: false)
        "tools": [                        # Only used when allow_tool_calls=true
            {"name": "memory_write", "description": "Write to memory"}
        ],
        "response_key": "summary",        # Store response in state.summary
        "stream": false,                  # Enable streaming (SSE only)
        "bypass_cache": false             # Skip idempotency cache (see §7)
    }
}
```

### 5.2 Message Templating

Messages support Jinja2-style state interpolation using `jinja2.sandbox.SandboxedEnvironment`. The sandboxing configuration:

- **Environment class**: `jinja2.sandbox.SandboxedEnvironment` — this is the only supported environment. `jinja2.Environment` is not used.
- **Disabled features**: `extensions=[]`, `undefined=jinja2.StrictUndefined` (unresolved vars raise errors rather than silently producing empty strings).
- **No imports**: `import` statements are blocked by the sandbox.
- **No file access**: Loader is `jinja2.BaseLoader()` — no filesystem access.
- **Attribute access**: Restricted to the template context only (`state`, `context`, `node_output`).

```
{{ state.field_name }}          → resolve from workflow state
{{ context.actor }}             → current actor
{{ context.run_id }}            → current run ID
{{ node_output.node_id.field }} → output from a specific node
```

#### 5.2.1 Second-Order Template Injection

LLM responses stored in workflow state can be referenced in subsequent node templates (e.g., `{{ state.llm_output }}`). This creates a **second-order injection path**: a model prompted with attacker-controlled input could produce output that, when interpolated into a later system prompt, injects instructions.

Mitigations:

1. **LLM output is marked untrusted in state**: Values written to `state[response_key]` by the LLM node handler carry an internal `_llm_output=true` metadata flag.
2. **Redaction filter applied**: LLM outputs pass through the existing redaction filter before storage.
3. **Template context does not expose `_` metadata keys**: Template rendering cannot access `state._*` keys.
4. **Operator guidance**: Workflow authors are responsible for prompt design. Documentation must warn against constructing system prompts that directly embed LLM outputs without human review. A future release will add a `sanitize_llm_output` node handler for this pattern.

> The sandbox prevents Python-level code execution. It does **not** prevent a crafted LLM string from functioning as a prompt injection when rendered into a subsequent LLM message. This is a fundamental limitation of the template approach and must be documented.

### 5.3 Response Handling

1. **Non-streaming**: Full response stored in `state[response_key]`. Includes `content`, `tool_calls` (if `allow_tool_calls=true`), `usage`, `latency_ms`.
2. **Streaming**: Chunks delivered via SSE to connected clients. Response stored after completion.

### 5.4 LLM-Originated Tool Calls

When `allow_tool_calls: true`, the model may respond with a `tool_calls` block. These calls are **not executed automatically**. The execution flow:

```
LLM Response with tool_calls
  │
  ▼
For each tool_call in response:
  1. Look up tool in tool registry (fail if not found → log + skip)
  2. RBAC check: does system:engine have tool:execute? (fail closed if not)
  3. Policy engine check: evaluate tool with actor=system:engine, context=workflow_run
      ├── DENY             → tool call skipped; reason recorded in state
      ├── REQUIRE_APPROVAL → approval request created; workflow pauses
      └── ALLOW            → tool executed
  4. Audit event: llm.tool_call with {node_id, tool_name, call_args, policy_decision}
  5. Tool result appended to messages for next LLM turn (if multi-turn is configured)
```

**Tool call arguments are treated as untrusted input** — they originate from LLM output and may contain injection attempts. Arguments are validated against the tool's declared parameter schema before execution. Validation failures are recorded as `llm.tool_call_invalid_args` audit events and the tool call is skipped.

**Default is `allow_tool_calls: false`**. Operators must explicitly opt in per node. This prevents tool execution via prompt injection in workflows not designed for tool-calling.

### 5.5 Retry Integration

LLM calls respect the node's `RetryPolicy`. Failures trigger exponential backoff. Specific provider errors are mapped:

| Provider Error | Behavior |
|----------------|----------|
| Rate limit (429) | Retry with backoff |
| Timeout | Retry with backoff |
| Auth error (401/403) | Fail immediately |
| Invalid model | Fail immediately |
| Server error (5xx) | Retry with backoff |

## 6. Streaming Support (SSE)

### 6.1 Endpoint

```
GET /api/v1/runs/{run_id}/stream
```

Returns Server-Sent Events. Authentication via streaming token (see §6.4).

### 6.2 Event Types

```
event: node_start
data: {"node_id": "summarize", "type": "ACTION", "timestamp": "..."}

event: llm_chunk
data: {"node_id": "summarize", "content": "Hello", "finish_reason": null, "timestamp": "..."}

event: llm_chunk
data: {"node_id": "summarize", "content": " world", "finish_reason": null, "timestamp": "..."}

event: llm_complete
data: {"node_id": "summarize", "usage": {"prompt_tokens": 42, "completion_tokens": 150}, "timestamp": "..."}
# NOTE: llm_complete does NOT embed the full response object to prevent unbounded event payloads.
# Clients that need the complete response must call GET /api/v1/runs/{run_id}/nodes/{node_id}
# after receiving llm_complete.

event: node_complete
data: {"node_id": "summarize", "status": "COMPLETED", "timestamp": "..."}

event: run_complete
data: {"run_id": "...", "status": "COMPLETED", "timestamp": "..."}
```

### 6.3 Connection Management

- Connections are tracked in a `ConnectionManager` (dict of `run_id` → set of connections).
- When a node executes, the engine broadcasts events to all connected clients for that run.
- Disconnected clients are cleaned up on next broadcast attempt.

**Reconnect recovery**: Clients that reconnect mid-run will miss events that fired while disconnected. To recover missed state:

```
GET /api/v1/runs/{run_id}/events?since=<ISO8601_timestamp>
```

This endpoint returns a paginated list of run events recorded after the given timestamp, ordered chronologically. It is backed by the audit log. Clients should call this endpoint on reconnect before resubscribing to the SSE stream to reconstruct any missed state.

### 6.4 Auth

The `run:read` permission is required.

**SSE authentication uses short-lived streaming tokens, not the actor's primary JWT.** The reason: browser-native `EventSource` cannot set custom headers, making query-parameter token passing unavoidable for browser clients. Putting a primary JWT in a query parameter exposes it in server logs, browser history, referrer headers on redirect, CDN logs, and error reporting tools — and the JWT grants all of the actor's permissions, not just `run:read`.

**Streaming token flow:**

```
1. Client calls POST /api/v1/runs/{run_id}/streaming-token
   - Requires valid Bearer token (primary JWT or API key)
   - Requires run:read permission
   - Returns {"streaming_token": "<opaque>", "expires_at": "<ISO8601>"}
   - Token is scoped to: this run_id only, run:read permission only
   - Token TTL: 5 minutes (configurable via SYNDICATECLAW_STREAMING_TOKEN_TTL_SECONDS)

2. Client calls GET /api/v1/runs/{run_id}/stream?token=<streaming_token>
   - Token is validated: not expired, correct run_id, run:read scope
   - Token is single-use: invalidated after first successful connection
   - On reconnect, client must request a new streaming token
```

**New endpoint:**

| Endpoint | Method | Required Permission |
|----------|--------|---------------------|
| `POST /api/v1/runs/{run_id}/streaming-token` | POST | `run:read` |

Streaming tokens are stored in the `streaming_tokens` table (see §10).

## 7. Idempotency

### 7.1 Idempotency Key Construction

The `idempotency_key` is constructed as:

```python
idempotency_key = f"{workflow_run_id}:{node_id}:{attempt_number}"
```

Where:
- `workflow_run_id` — the current workflow run's ID
- `node_id` — the ID of the `llm` node
- `attempt_number` — the current retry attempt (0-indexed)

**Implications:**

- The same node in the same run on the same attempt is deduplicated (the primary use case: re-entrant execution during recovery).
- A **retry** (attempt 1, 2, ...) gets a new idempotency key and calls the provider fresh. This is intentional: retries happen because attempt N failed; returning a cached response from the failed attempt would not help.
- A **replay** of a completed run uses `attempt_number=0` on all nodes, which means the first LLM call on each node would return a cached response (if TTL has not expired). If a fresh response is needed on replay, set `bypass_cache: true` on the node config, or use the `bypass_cache` field in the API.

### 7.2 Request Hashing

The request hash covers all parameters that affect the response:

```python
hash_input = json.dumps({
    "provider": provider,
    "model": model,
    "messages": messages,
    "temperature": temperature,
    "max_tokens": max_tokens,
    "tools": tool_defs,
}, sort_keys=True)
request_hash = sha256(hash_input.encode()).hexdigest()
```

### 7.3 Idempotency Store

The `idempotency_records` table stores:

| Column | Type | Description |
|--------|------|-------------|
| `idempotency_key` | TEXT | `{run_id}:{node_id}:{attempt}` |
| `request_hash` | TEXT | SHA-256 of the request |
| `provider` | TEXT | Provider name |
| `model` | TEXT | Model name |
| `response` | JSONB | Cached response |
| `usage` | JSONB | Token usage |
| `created_at` | TIMESTAMP | When the call was made |
| `expires_at` | TIMESTAMP | TTL for the record |

**Default TTL**: 24 hours (`SYNDICATECLAW_LLM_IDEMPOTENCY_TTL_SECONDS`, default `86400`).

A cleanup cron job (`syndicateclaw.tasks.idempotency_cleanup`) runs every hour and deletes records where `expires_at < now()`.

### 7.4 Deduplication Flow

1. If `bypass_cache=true` on the node config, skip steps 2–3 entirely and call the provider.
2. Compute `idempotency_key` and `request_hash`.
3. Check `idempotency_records` for a matching `(idempotency_key, request_hash)` where `expires_at > now()`.
4. If found, return the cached response (audit event: `llm.cache_hit`).
5. If not found, call the provider and store the response with `expires_at = now() + TTL`.

## 9. Security

### 9.1 API Key Storage

Provider API keys are stored in environment variables, never in the database or workflow definitions. The provider config references keys via `${ENV_VAR}` syntax. In production, these environment variables must be populated from a secrets manager — not from `.env` files, CI/CD plaintext variables, or inline YAML values.

### 9.2 SSRF Prevention

All provider API calls go through the HTTP client with SSRF protection enabled. Provider hostnames are validated against the allowlist in `providers.yaml`.

The allowlist validation is enforced in the adapter layer, not solely in the config loader. Even if an operator modifies the host after startup, each HTTP call validates the resolved IP against the SSRF block list (`syndicateclaw.security.ssrf.validate_url()`).

### 9.3 Provider Test Endpoint Security

`POST /api/v1/providers/{name}/test` tests provider connectivity.

- **Required permission**: `admin:*`. This endpoint is for operator diagnostics only.
- **SSRF protection**: The endpoint calls the provider via the same HTTP client and SSRF validation as all other provider calls. Attempts to test a provider with a private/internal host are rejected with the SSRF error reason.
- **Error responses**: On failure, the endpoint returns a generic `{"status": "unreachable"}` with an error code. It does **not** include resolved IP addresses, internal DNS details, or network topology in the response body.

### 9.4 Prompt Injection

LLM responses are treated as untrusted input. See §5.4 for tool call gating. See §5.2.1 for second-order template injection mitigations. Workflow state values written by LLM node handlers are tagged with `_llm_output=true` to enable downstream redaction and sanitization.

## 13. RBAC Coverage for LLM Invocations

LLM node handler calls execute server-side within workflow runs — they are not triggered by a direct HTTP request from a user. The RBAC route registry in v1.1.0 covers the HTTP surface (`POST /api/v1/workflows/{id}/runs` requires `run:create`). Internal workflow execution is not HTTP-gated.

The actor for LLM node execution is the workflow engine's service account (`system:engine`). This account must have:

- `run:control` — to update run and node execution state
- `tool:execute` — to invoke tools on behalf of LLM tool calls (when `allow_tool_calls=true`)
- Any permissions required by tools called via LLM tool-calling

`system:engine` must be explicitly configured with these permissions in the RBAC system before v1.2.0 deploys. The deployment runbook must include this step.

For `allow_tool_calls=true` nodes: the policy engine evaluates tool calls with `actor=system:engine`. Workflow authors who want per-user policy enforcement on LLM tool calls should design workflows to pass the originating actor's identity in the workflow context and write policy rules that inspect `context.initiated_by`.

- `test_sse_reconnect_recovery` — `GET /api/v1/runs/{id}/events?since=` returns missed events after simulated disconnect
- `test_sse_streaming_token_required` — Direct JWT in query param returns 401; streaming token required
- `test_provider_test_endpoint_requires_admin` — Non-admin actor receives 403 on `POST /api/v1/providers/{name}/test`
- `test_provider_test_endpoint_ssrf_blocked` — Provider configured with private host returns SSRF error, not network details

### 14.4 CI Secret Management

Integration tests that require real provider API keys (`test_llm_handler_calls_openai`) are marked `@pytest.mark.requires_api_keys` and are **skipped by default in CI**. They run only when `SYNDICATECLAW_CI_PROVIDER_TESTS=true` is set, with keys injected via CI secrets:

```yaml
provider_integration_tests:
  stage: integration
  when: manual  # or: only on release branches
  variables:
    SYNDICATECLAW_CI_PROVIDER_TESTS: "true"
  secrets:
    OPENAI_API_KEY:
      vault: ci/syndicateclaw/openai_api_key@secret
    ANTHROPIC_API_KEY:
      vault: ci/syndicateclaw/anthropic_api_key@secret
  script:
    - pytest tests/ -m requires_api_keys -q --tb=short
```

Never hardcode API keys in CI YAML or test fixture files.

## 16. Risks

| Risk | Mitigation |
|------|------------|
| Provider API changes | Adapter protocol isolates provider-specific logic; unit tests per adapter |
| Prompt injection via tool-calling | `allow_tool_calls` defaults to `false`; tool calls gated by policy engine; arg validation against schema |
| Second-order template injection | LLM outputs tagged and documented; sandbox prevents Python execution but not prompt injection |
| Streaming token leakage | Short TTL (5 min); single-use; scoped to run_id and run:read only |
| Idempotency cache returning stale responses on replay | `bypass_cache` flag; attempt-number-keyed idempotency; documented replay behavior |
| Token cost overrun | `max_tokens` enforced per call; per-run budget deferred to enterprise tier |
| Idempotency table growth | TTL-based expiration (24h default); hourly cleanup cron job |
| Missing provider key at startup | Fatal startup error on missing env var; no silent partial initialization |
