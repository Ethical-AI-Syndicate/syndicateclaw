# v1.2.0 — LLM Ready

**Sprint Duration:** 4 weeks
**Goal:** Enable real LLM invocation through the workflow engine with streaming support

---

## 1. Problem Statement

SyndicateClaw's `llm` node handler is a placeholder — it annotates state but doesn't call any model. This makes the platform unusable for LLM-powered workflows, which is the primary use case for agent orchestration. Without native LLM support, users must build custom tool handlers for every model invocation, losing the benefits of the platform's audit, policy, and retry infrastructure.

## 2. Goals

- **Native LLM invocation**: `llm` node handlers call real models via a provider abstraction
- **Provider abstraction**: Support OpenAI, Anthropic, Azure, and Ollama through a unified interface
- **Streaming**: LLM responses can be streamed to clients via SSE
- **Observability**: LLM-specific metrics and tracing for model calls
- **Idempotency**: LLM calls are deduplicated by request hash

## 3. Non-Goals

- Multi-agent messaging (v1.3.0)
- Visual workflow builder (v1.5.0)
- Fine-tuning or model management
- Cost tracking per-call (deferred to enterprise tier)

## 4. Provider Abstraction Layer

### 4.1 Design

A `ProviderService` mediates all model calls:

```
Workflow Engine → llm node handler → ProviderService → ProviderAdapter → Model API
```

**ProviderAdapter protocol:**

```python
class ProviderAdapter(Protocol):
    async def complete(
        self,
        messages: list[Message],
        model: str,
        tools: list[Tool] | None = None,
        stream: bool = False,
        **kwargs: Any,
    ) -> CompletionResponse | AsyncIterator[StreamChunk]:
        ...

    async def embed(
        self,
        texts: list[str],
        model: str,
    ) -> list[list[float]]:
        ...
```

**Supported providers:**

| Provider | Env Var | Models | Notes |
|----------|---------|--------|-------|
| OpenAI | `OPENAI_API_KEY` | gpt-4o, gpt-4o-mini, o1 | Via openai-python |
| Anthropic | `ANTHROPIC_API_KEY` | claude-sonnet-4-20250514, claude-haiku-4-20250414 | Via anthropic-python |
| Azure | `AZURE_OPENAI_*` | Azure-hosted models | Via openai-python |
| Ollama | `OLLAMA_HOST` | Local models | Via ollama-python |

### 4.2 Provider Config

`providers.yaml` (loaded at startup):

```yaml
providers:
  openai:
    type: openai
    api_key: ${OPENAI_API_KEY}
    models:
      - gpt-4o
      - gpt-4o-mini
  anthropic:
    type: anthropic
    api_key: ${ANTHROPIC_API_KEY}
    models:
      - claude-sonnet-4-20250514
      - claude-haiku-4-20250414
  ollama:
    type: ollama
    host: ${OLLAMA_HOST:-http://localhost:11434}
    models:
      - llama3
      - codellama

routing:
  default_provider: openai
  rules:
    - match: "model:*"
      provider: openai
    - match: "model:claude*"
      provider: anthropic
    - match: "model:llama*"
      provider: ollama
```

## 5. LLM Node Handler

### 5.1 Node Config

```python
{
    "id": "summarize",
    "type": "ACTION",
    "handler": "llm",
    "config": {
        "provider": "openai",         # Optional, auto-routes if omitted
        "model": "gpt-4o",
        "messages": [
            {"role": "system", "content": "You are a helpful assistant."},
            {"role": "user", "content": "{{ state.input_text }}"}
        ],
        "temperature": 0.7,
        "max_tokens": 1024,
        "tools": [                    # Optional, for tool-calling
            {"name": "memory_write", "description": "Write to memory"}
        ],
        "response_key": "summary",    # Store response in state.summary
        "stream": false               # Enable streaming (SSE only)
    }
}
```

### 5.2 Message Templating

Messages support Jinja2-like state interpolation:

```
{{ state.field_name }}          → resolve from workflow state
{{ context.actor }}             → current actor
{{ context.run_id }}            → current run ID
{{ node_output.node_id.field }} → output from a specific node
```

The template engine is sandboxed (no imports, no file access).

### 5.3 Response Handling

1. **Non-streaming**: Full response stored in `state[response_key]`. Includes `content`, `tool_calls`, `usage`, `latency_ms`.
2. **Streaming**: Chunks delivered via SSE to connected clients. Response stored after completion.

### 5.4 Retry Integration

LLM calls respect the node's `RetryPolicy`. Failures trigger exponential backoff. Specific provider errors are mapped:

| Provider Error | Behavior |
|----------------|----------|
| Rate limit (429) | Retry with backoff |
| Timeout | Retry with backoff |
| Auth error (401/403) | Fail immediately |
| Invalid model | Fail immediately |
| Server error (5xx) | Retry with backoff |

## 6. Streaming Support (SSE)

### 6.1 Endpoint

```
GET /api/v1/runs/{run_id}/stream
```

Returns Server-Sent Events. Authentication via Bearer token or query parameter.

### 6.2 Event Types

```
event: node_start
data: {"node_id": "summarize", "type": "ACTION", "timestamp": "..."}

event: llm_chunk
data: {"node_id": "summarize", "content": "Hello", "finish_reason": null, "timestamp": "..."}

event: llm_chunk
data: {"node_id": "summarize", "content": " world", "finish_reason": null, "timestamp": "..."}

event: llm_complete
data: {"node_id": "summarize", "response": {...}, "usage": {"prompt_tokens": 42, "completion_tokens": 150}, "timestamp": "..."}

event: node_complete
data: {"node_id": "summarize", "status": "COMPLETED", "timestamp": "..."}

event: run_complete
data: {"run_id": "...", "status": "COMPLETED", "timestamp": "..."}
```

### 6.3 Connection Management

- Connections are tracked in a `ConnectionManager` (dict of run_id → set of connections).
- When a node executes, the engine broadcasts events to all connected clients for that run.
- Disconnected clients are cleaned up on next broadcast attempt.
- No persistence — if you reconnect, you only get events from that point forward.

### 6.4 Auth

- SSE endpoint requires a valid JWT or API key.
- The `runs:read` permission is required.
- Query parameter `?token=<jwt>` supported for browser EventSource (which can't set headers).

## 7. Idempotency

### 7.1 Request Hashing

LLM calls are hashed for deduplication:

```python
hash_input = json.dumps({
    "provider": provider,
    "model": model,
    "messages": messages,
    "temperature": temperature,
    "max_tokens": max_tokens,
    "tools": tool_defs,
}, sort_keys=True)
request_hash = sha256(hash_input)
```

### 7.2 Idempotency Store

The `idempotency_records` table stores:

| Column | Type | Description |
|--------|------|-------------|
| `idempotency_key` | TEXT | Workflow-scoped dedup key |
| `request_hash` | TEXT | SHA-256 of the request |
| `provider` | TEXT | Provider name |
| `model` | TEXT | Model name |
| `response` | JSONB | Cached response |
| `usage` | JSONB | Token usage |
| `created_at` | TIMESTAMP | When the call was made |
| `expires_at` | TIMESTAMP | TTL for the record |

### 7.3 Deduplication Flow

1. Before calling the provider, compute `request_hash`.
2. Check `idempotency_records` for a matching `(idempotency_key, request_hash)`.
3. If found and not expired, return the cached response (audit event: `llm.cache_hit`).
4. If not found, call the provider and store the response.
5. `idempotency_key` is derived from the workflow run context (ensures same request in same run is deduped).

## 9. Security

### 9.1 API Key Storage

Provider API keys are stored in environment variables, never in the database or workflow definitions. The provider config references keys via `${ENV_VAR}` syntax.

### 9.2 SSRF Prevention

All provider API calls go through the HTTP client with SSRF protection enabled. Provider hostnames are validated against the allowlist in `providers.yaml`.

### 9.3 Prompt Injection

LLM responses are treated as untrusted input. When stored in workflow state, they go through the redaction filter. When used as input to subsequent nodes, they're subject to the same validation as any other state field.

- `test_idempotency_different_run_no_dedup` — Same request in different run calls provider

## 15. Risks

| Risk | Mitigation |
|------|------------|
| Provider API changes | Adapter protocol isolates provider-specific logic; unit tests per provider |
| Streaming complexity | Start with non-streaming; streaming is additive |
| Token cost overrun | `max_tokens` enforced per call; can add per-run budget in future |
| Idempotency table growth | TTL-based expiration; cleanup cron job |

## 16. Rollback

- `llm` node handler: revert to placeholder behavior (annotate state only)
- SSE endpoint: remove or return 501
- Provider config: remove file reference; platform works without LLM
