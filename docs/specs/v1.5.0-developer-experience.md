## 1. Problem Statement

SyndicateClaw's API is comprehensive but has a high barrier to entry: workflows are defined in raw JSON, there is no Python SDK, and there is no way to visualize workflow graphs. Developers must read API docs, manually construct requests, and debug via HTTP responses. This limits adoption and slows iteration.

---

## 2. Goals

- **Python SDK**: Typed, Pythonic client for all API endpoints with streaming token integration
- **Visual workflow builder**: Web UI to create and edit workflows with secure embedding
- **Plugin system**: Extensible architecture for custom lifecycle hooks with enforced sandbox

---

## 3. Non-Goals

- Mobile SDK (future)
- JavaScript/TypeScript SDK (future)
- Marketplace for plugins (future)
- Workflow templates (future)
- File-path plugin loading (removed as security risk — see §6.2)

## 4. Python SDK

### 4.1 Package

```
pip install syndicateclaw-sdk
```

### 4.2 Client Design

```python
from syndicateclaw_sdk import SyndicateClaw

# Initialize with API key
client = SyndicateClaw(
    base_url="https://syndicateclaw.example.com",
    api_key="sk-...",
    min_server_version="1.5.0",  # Checked at init; raises IncompatibleServerError if server < this
)

# Or with JWT token
client = SyndicateClaw.from_token(
    base_url="https://syndicateclaw.example.com",
    token="eyJ...",
    min_server_version="1.5.0",
)
```

At construction, the SDK calls `GET /api/v1/info` to retrieve the server version and validates `server_version >= min_server_version`. If the check fails, `IncompatibleServerError` is raised with a message indicating the required and actual versions. This prevents silent breakage when an SDK is used against an older server that lacks endpoints it expects.

### 4.3 API Methods

```python
# Workflows
workflow = client.workflows.create(
    name="my-workflow",
    nodes=[...],
    edges=[...]
)
workflow = client.workflows.get("wf-abc123")
workflows = client.workflows.list(status="active")
client.workflows.delete("wf-abc123")

# Runs
run = client.workflows.runs.create("wf-abc123", input_state={"key": "value"})
run = client.workflows.runs.get("wf-abc123", "run-xyz789")
client.workflows.runs.pause("wf-abc123", "run-xyz789")
client.workflows.runs.resume("wf-abc123", "run-xyz789")

# Streaming — transparently handles streaming token acquisition and rotation
async for event in client.workflows.runs.stream("wf-abc123", "run-xyz789"):
    print(event.type, event.data)

# Memory — all calls include namespace context
client.memory.write(namespace="my-namespace", key="key1", value={"value": "data"})
record = client.memory.read(namespace="my-namespace", key="key1")
results = client.memory.search(namespace="my-namespace", query="data")

# Tools
client.tools.execute("http_request", {"url": "https://api.example.com"})

# Agents
agent = client.agents.register(name="my-agent", capabilities=["research"])
client.agents.heartbeat(agent.id)  # Uses ID, not name
msg = client.messages.send(recipient_id="01ABCDEF...", content={"action": "do_something"})
reply = client.messages.reply(msg.id, content={"result": "done"})

# Approvals
pending = client.approvals.list(status="PENDING")
client.approvals.approve("approval-abc123")
client.approvals.reject("approval-abc123", reason="Insufficient evidence")

# Organizations
org = client.organizations.create(name="acme-corp")
client.organizations.add_member("org-abc123", actor="user@example.com", role="MEMBER")
```

### 4.4 Streaming Token Integration

The SDK handles streaming token lifecycle transparently. The `stream()` method:

1. Calls `POST /api/v1/runs/{run_id}/streaming-token` to obtain a short-lived token (scoped to this run and `run:read` only).
2. Opens the SSE connection with `?token=<streaming_token>`.
3. On SSE reconnect, automatically fetches a new streaming token before reconnecting.
4. On token expiry (server returns 401), fetches a new token and reconnects.

The primary API key/JWT is never placed in a URL query parameter.

### 4.5 Workflow Builder (Fluent API)

```python
from syndicateclaw_sdk.builder import WorkflowBuilder, BuildValidationError

wf = (
    WorkflowBuilder("invoice-approval")
    .description("Automated invoice approval with human review")
    .start("init")
    .action("validate", handler="validate_invoice")
    .decision(
        "high_amount",
        condition="state.invoice_amount > 5000",
        true_branch=lambda b: b.approval("manual_review", assigned_to=["finance@company.com"]),
        false_branch=lambda b: b.action("auto_approve", handler="approve_invoice")
    )
    .checkpoint("after_approval")
    .end("complete")
    .build()  # Raises BuildValidationError if validation fails
)

client.workflows.create_from_builder(wf)
```

**Validation at `.build()` time** (raises `BuildValidationError` with descriptive message):
- Every `decision()` call must supply both `true_branch` and `false_branch`. Missing branch raises `BuildValidationError("Decision node 'X' is missing true_branch/false_branch")`.
- Condition strings are validated against the safe expression evaluator grammar. Invalid conditions raise `BuildValidationError("Invalid condition expression: ...")`.
- The resulting definition is validated against the workflow definition JSON schema.
- All node IDs are unique within the workflow.
- Graph has exactly one START node and at least one END node.
- No disconnected subgraphs.

### 4.6 Local Development Mode

```python
from syndicateclaw_sdk.local import LocalRuntime

# In-memory execution — explicitly no database, no policy engine, no audit
runtime = LocalRuntime()
result = runtime.execute(wf, input_state={"invoice_amount": 3000})
print(result.state)
```

**Security controls absent in LocalRuntime** (printed as a WARNING at construction time in non-production environments, and as a logged ERROR if somehow constructed in production):

| Control | Status in LocalRuntime | Impact |
|---------|----------------------|--------|
| Policy engine | Absent | All tools are allowed; no DENY decisions |
| Audit service | Absent | No audit trail for local executions |
| RBAC | Absent | No permission checks |
| Approval system | Absent | `APPROVAL` nodes auto-approve |
| Tool executor sandbox | Absent | No SSRF or payload checks on tool calls |
| Database persistence | Absent | Run state is in-memory only |

**Production guard**: If `SYNDICATECLAW_ENVIRONMENT` is not one of `development`, `dev`, `test`, `testing`, `LocalRuntime()` raises `RuntimeError("LocalRuntime cannot be constructed in production environments")`.

**Documentation requirement**: SDK docs must prominently warn that workflows tested with LocalRuntime may behave differently in production due to missing policy, approval, and RBAC controls. A dedicated "Testing vs. Production" guide is required.

### 4.7 Async Support

The SDK uses an `httpx.AsyncClient` internally. All methods are async-first:

```python
# Async (primary interface)
result = await client.workflows.create(...)
async for event in client.workflows.runs.stream(...):
    ...

# Sync convenience wrappers (uses asyncio.run() internally)
result = client.workflows.create_sync(...)
```

This is the reverse of the original spec's pattern (sync-primary with `async_*` variants). Async-first is the standard for modern Python clients interacting with async backends.

### 4.8 Error Handling

```python
from syndicateclaw_sdk.exceptions import (
    WorkflowNotFoundError,      # 404
    ToolDeniedError,            # 403 from policy engine
    ApprovalRequiredError,      # 202 from approval gate
    RateLimitError,             # 429
    AuthenticationError,        # 401
    QuotaExceededError,         # 429 with quota details
    IncompatibleServerError,    # Server version < min_server_version
    BuildValidationError,       # WorkflowBuilder validation failure
)

try:
    client.tools.execute("http_request", {"url": "https://blocked.internal"})
except ToolDeniedError as e:
    print(f"Tool denied: {e.reason}")
except RateLimitError as e:
    print(f"Rate limited. Retry after {e.retry_after}s")
```

## 5. Visual Workflow Builder

### 5.1 Architecture

A React-based web UI that:
1. Renders workflow graphs visually (using react-flow — see §5.7 for license note)
2. Supports drag-and-drop node creation
3. Allows edge editing and condition configuration
4. Imports/exports JSON workflow definitions
5. Communicates with the SyndicateClaw API using scoped builder tokens (not primary JWTs)

### 5.2 Components

| Component | Description |
|-----------|-------------|
| `WorkflowCanvas` | Main graph canvas (react-flow) |
| `NodePalette` | Drag-and-drop node types (START, END, ACTION, DECISION, APPROVAL, CHECKPOINT) |
| `NodeConfig` | Side panel for configuring selected node |
| `EdgeConfig` | Side panel for configuring selected edge |
| `StatePreview` | Live preview of workflow state |
| `RunControls` | Start/pause/resume/cancel buttons |

### 5.3 Secure Embedding

The builder is served as a standalone React app. When embedded via iframe, it uses a **scoped builder token** — not the primary JWT — to prevent credential leakage via iframe `src` URL:

```html
<!-- Step 1: Server-side — issue a builder token -->
<!-- POST /api/v1/workflows/{id}/builder-token → {"builder_token": "...", "expires_at": "..."} -->
<!-- Token is scoped to: workflow_id, workflow:read + workflow:update permissions, 1-hour TTL -->

<!-- Step 2: Embed using the builder token -->
<iframe
  src="https://syndicateclaw.example.com/builder/{workflow_id}?token={builder_token}"
  width="100%"
  height="800px"
></iframe>
```

**Builder token properties:**
- Scoped to the specific `workflow_id` only
- Grants `workflow:read` and `workflow:update` on that workflow only
- TTL: 1 hour (configurable via `SYNDICATECLAW_BUILDER_TOKEN_TTL_SECONDS`)
- Single-session: token is bound to the session that opened the builder; cannot be shared

**New endpoint:**

| Endpoint | Method | Required Permission | Notes |
|----------|--------|---------------------|-------|
| `POST /api/v1/workflows/{id}/builder-token` | POST | `workflow:manage` | Issues scoped builder token |

### 5.4 CSRF Protection

The builder calls `PUT /api/v1/workflows/{id}` to save. To prevent CSRF attacks when the builder is iframe-embedded:

- All state-modifying requests from the builder include a `X-Builder-Token` header (the scoped builder token).
- The server validates that the token in the header matches a valid builder session for the workflow.
- Cross-origin POSTs from malicious embedders cannot set the `X-Builder-Token` header (CORS blocks them from reading the token value).
- The builder never stores the builder token in cookies (preventing cookie-based CSRF).

### 5.5 API Integration

The builder communicates with the existing API using the builder token for authentication:

- `GET /api/v1/workflows/{id}` — Load workflow definition
- `PUT /api/v1/workflows/{id}` — Save workflow definition
- `POST /api/v1/workflows/{id}/runs` — Start a run (requires re-authentication with primary credentials if `workflow:run` is not in builder token scope)
- `GET /api/v1/runs/{id}/stream` — Stream events (acquires a streaming token transparently)

### 5.6 Code View

Toggle between visual and code views:

- **Visual**: react-flow graph with node/edge configuration panels
- **Code**: Raw JSON editor (Monaco editor for syntax highlighting)
- **Diff**: Compare current edit with saved version (uses `GET /api/v1/workflows/{id}/diff`)

### 5.7 react-flow License Note

react-flow is MIT-licensed for open source use. Commercial deployments of the builder must verify license compliance with the react-flow commercial terms before public release. This must be resolved before v1.5.0 ships.

### 5.8 Endpoints

| Endpoint | Method | Change | Required Permission |
|----------|--------|--------|---------------------|
| `GET /builder/{workflow_id}` | GET | **New** — Serve builder with workflow loaded | Any authenticated principal with `workflow:read` |

## 6. Plugin System

### 6.1 Plugin Interface

```python
from syndicateclaw.plugins import Plugin, PluginContext

class MyPlugin(Plugin):
    name = "my-plugin"
    version = "1.0.0"

    async def on_workflow_start(self, ctx: PluginContext) -> None:
        """Called before workflow execution begins. State is read-only."""
        pass

    async def on_node_execute(self, ctx: PluginContext, node_id: str, result: Any) -> None:
        """Called after each node executes AND after its audit event is written."""
        pass

    async def on_workflow_end(self, ctx: PluginContext, status: str) -> None:
        """Called after workflow execution completes."""
        pass

    async def on_error(self, ctx: PluginContext, error: Exception) -> None:
        """Called on workflow error."""
        pass
```

### 6.2 Plugin Loading

Plugins are loaded **only** from Python package entry points — no file-path loading.

```python
# In your plugin package's pyproject.toml:
[project.entry-points."syndicateclaw.plugins"]
my-plugin = "my_plugin_package:MyPlugin"
```

```yaml
# plugins.yaml — references entry point names or fully-qualified class paths from installed packages
plugins:
  - syndicateclaw.plugins.builtin.AuditTrailPlugin
  - syndicateclaw.plugins.builtin.MetricsPlugin
  - my_plugin_package:MyPlugin  # From installed package only; not a file path
```

**Why file-path loading is prohibited**: Loading a plugin from an arbitrary filesystem path executes untrusted Python code as the service process. This contradicts the platform's core architecture principle of **Explicitness over Convenience** (no auto-discovery, no dynamic loading). Entry-point plugins are installed packages with pinned versions, reviewable source, and audit trails. File-path plugins have none of these properties.

**Entry point validation at startup**: The plugin registry validates that all configured entry points resolve to a class that:
1. Inherits from `Plugin`
2. Has `name` and `version` class attributes
3. Implements at least one hook method

Unknown or unresolvable entry points cause a **fatal startup error** (not a warning).

### 6.3 Plugin Context and Sandbox

```python
class PluginContext:
    run_id: str
    workflow_id: str
    actor: str
    namespace: str

    @property
    def state(self) -> types.MappingProxyType:
        """Read-only view of current workflow state.
        
        Returns a MappingProxyType — attempts to modify raise TypeError:
            ctx.state["key"] = "value"  # Raises: TypeError: 'mappingproxy' object does not support item assignment
            ctx.state.update({...})     # Raises: AttributeError: 'mappingproxy' has no attribute 'update'
        """
        return types.MappingProxyType(self._state_snapshot)
```

**Sandbox enforcement:**
- `ctx.state` returns `types.MappingProxyType(copy.deepcopy(state_snapshot))` — a deep copy wrapped in a read-only proxy. Plugins receive a copy; they cannot mutate the live state even if they somehow bypass the proxy (e.g., via `object.__setattr__`).
- Plugins cannot call `asyncio.create_task()`, `threading.Thread()`, or `subprocess` — these escape the timeout. Plugin code is inspected at registration for banned imports and calls (`ast.walk` check at load time). Plugins containing `asyncio.create_task`, `threading`, `subprocess`, `os.system`, or `importlib` in their source are rejected at startup with a `PluginSecurityViolationError`.
- Timeout enforcement uses `asyncio.wait_for(plugin.hook(...), timeout=PLUGIN_TIMEOUT)`. Tasks spawned within the hook are not awaited by the timeout boundary, so spawning is prohibited at the source level (see above).

### 6.4 Plugin Execution Order and Audit

Plugin hooks are executed **after** the workflow engine's own audit event for that lifecycle point is written. This ensures the audit record exists before the plugin can observe or react to it.

Execution order for `on_node_execute`:
1. Node handler executes.
2. `NodeExecution.status` updated in DB.
3. Audit event `NODE_EXECUTION_COMPLETED` written.
4. `on_node_execute` plugin hooks called (in registration order).
5. Plugin audit events written (see §6.5).

If a plugin hook raises an exception, it is caught, logged as `plugin.hook_failed plugin_name=<> hook=<> error=<>`, and execution continues with the next plugin. Plugin errors never propagate to the workflow engine.

### 6.5 Plugin Audit Events

Plugin actions are audited with these event types:

| Event Type | When Emitted |
|------------|-------------|
| `plugin.hook_invoked` | Before each hook call |
| `plugin.hook_completed` | After successful hook return |
| `plugin.hook_failed` | When hook raises exception |
| `plugin.hook_timeout` | When hook exceeds `SYNDICATECLAW_PLUGIN_TIMEOUT` |
| `plugin.security_violation` | When plugin is rejected at load time |

### 6.6 Built-in Plugins

| Plugin | Description | Notes |
|--------|-------------|-------|
| `AuditTrailPlugin` | Enhanced audit logging with custom fields | |
| `CostTrackingPlugin` | Track LLM token costs per workflow | |
| `RateLimitPlugin` | Custom rate limiting rules | |
| `WebhookPlugin` | Send webhooks on workflow events | All webhook URLs validated via `syndicateclaw.security.ssrf.validate_url()` before each send |
| `MetricsPlugin` | Custom Prometheus metrics | |

**`WebhookPlugin` SSRF protection**: Webhook destination URLs are validated via the platform's existing `validate_url()` on every request, not just at registration. URL redirects are not followed (httpx `follow_redirects=False`). Webhook URLs must be HTTPS in non-development environments.

### 6.7 Plugin Discovery and Loading

Plugins are loaded from installed Python packages only:

1. **Built-in**: `syndicateclaw.plugins.builtin` (always available)
2. **Package**: `pip install syndicateclaw-plugin-*` — auto-discovered via `importlib.metadata.entry_points(group="syndicateclaw.plugins")`. Package version must be pinned in deployment requirements.

```yaml
# plugins.yaml
plugins:
  - syndicateclaw.plugins.builtin.AuditTrailPlugin
  - syndicateclaw.plugins.builtin.MetricsPlugin
  - my_company_plugin:CompliancePlugin  # From installed package
```

## 7. Database Migrations and Schema Changes

This release introduces no new database tables. However:

- Plugin audit events use the existing `audit_events` table. Plugin-specific event types (`plugin.hook_invoked`, etc.) are added to the allowed `event_type` enum. If `event_type` is a CHECK constraint, migration `024_plugin_event_types` adds the new values.
- Builder tokens use the existing `streaming_tokens` table with `token_type: "builder"` discriminator (added via `ALTER TABLE streaming_tokens ADD COLUMN token_type TEXT NOT NULL DEFAULT 'streaming'`). Migration: `025_builder_token_type`.

| Migration | Table | Change |
|-----------|-------|--------|
| `024_plugin_event_types` | `audit_events` | Add plugin event type values to `event_type` enum (if constrained) |
| `025_builder_token_type` | `streaming_tokens` | Add `token_type TEXT NOT NULL DEFAULT 'streaming'` |

## 11. Risks

| Risk | Mitigation |
|------|------------|
| Plugin sandbox escape via reflection | AST check blocks known patterns; deep copy + proxy prevents state mutation; subprocess/thread spawn blocked |
| Builder token leakage | Short TTL (1h); single-session binding; never in primary JWT |
| LocalRuntime misuse in production | Production guard raises `RuntimeError`; prominent documentation warning |
| react-flow license for commercial use | Legal review required before v1.5.0 ships; may need license purchase or alternative library |
| Entry point supply chain attack | Packages must be pinned in deployment requirements; reviewed in v2.0.0 `pip-audit` scan |
| SDK API drift with server | Server version check at init; auto-generation from OpenAPI spec as long-term solution |

---

## 12. Rollback

- **SDK**: Remove PyPI package; existing API clients unaffected.
- **Builder**: Disable endpoint (`SYNDICATECLAW_BUILDER_ENABLED=false`); builder token table column persists but unused.
- **Plugins**: Disable plugin loading (`SYNDICATECLAW_PLUGINS_FILE=` unset); core functionality unaffected.
